# 1. この章でやること
この章は「[Homebrew](https://brew.sh/ja/) や [mise](https://github.com/jdx/mise) と競合しないか不安な人向け」の解説です。

また、Nix と各種パッケージマネージャーをどう使い分けるかについても紹介します。


# 2. 競合と共存に向けた考え方
同じツールが複数の仕組みからインストールされた場合、**どこからインストールされたツールを使うかは PATH（実行ファイルの探索順）で決まります**。

Nix・Homebrew・mise のいずれもパッケージのインストール先（バイナリの管理方法）が異なるため、**共存自体は可能**です。
しかし、同じツールを複数で管理すると、「どれが使われているかわからない」状態になりがちです。

そのため、**各マネージャーの役割分担を明確にしておくことを推奨します**。
**また、Nix にも強み・弱みがあるため、適度に他のパッケージマネージャーと併用すべきです**。


:::details どのパッケージマネージャー管理下かを確認する方法
以下のコマンドで確認できます。

```zsh:Zsh
# どのパスのツールが使われているか確認
which git

# 重複してインストールされているか確認
which -a git
```

実行例。

```zsh:Zsh
> which git
/etc/profiles/per-user/ryu/bin/git

> which -a git    
/opt/homebrew/bin/git
/etc/profiles/per-user/ryu/bin/git
/usr/bin/git
```

:::


# 3. Homebrew と Nix の共存
Homebrew は Mac 向けの GUI アプリや CLI ツールが豊富なパッケージマネージャーです。

一方 Nix では、商用の Mac 用 GUI アプリなどは利用できないことが多いです。

:::message
ビルドの再現性を重視する Nix の仕様上、「ソースコードが公開されていないもの」は対応外となりがちです。
:::

**そのため、Mac においては Homebrew と併用することが事実上マストになります**。

次の方針にするとシンプルになるかと思います。

- Nix で一般的なパッケージを揃える（開発に利用するツールなら大抵は揃っています）
- Homebrew で Mac 専用のアプリや GUI アプリ（Cask）を管理する

この分担にしておけば、Homebrew の強みを活かしつつ、Nix の利点（再現性）を享受できます。


# 4. mise と Nix の共存
mise は言語ランタイムや CLI ツールのバージョン管理を手軽に行えるのが強みです。
`.mise.toml` で特定ディレクトリでのツールバージョンを固定するのに加えて、タスクランナーや環境変数の設定機能を利用している方も多いかもしれません。

Nix でも同様のことができるため、**役割が重なりやすい**です。

うまく両者を併用しつつ、役割を明確に分けることを推奨します。

- 例 1
  - 言語ランタイムの管理は mise
  - それ以外のツールの管理は Nix
  - 環境変数やタスクの管理は mise
- 例 2
  - ツールの管理は Nix
  - 環境変数やタスクの管理は mise


:::message
Node.js、Python、Terraform といったバージョンを指定・切り替えしたいツールの場合、Nix では実現が手間です（書き方を学ぶ必要があります）。
ですので、これらを mise で管理するのはアリだと思います。

----

また、タスクランナーや環境変数の管理といった機能も、mise やその他のツールで管理するのが楽だと思います。
Nix で実現する場合、独自の書き方（正確には、Flakes の devShell という機能）を学ぶ必要があり、敷居が高くなります。

個人的には、Nix はパッケージの管理に専念させ、それ以外は他のツールで補うのをおすすめします。

>※筆者個人の好みとして、1 つのツールに責務を過剰に持たせることを好まない（ツールへの依存が強まりツール変更が困難になる、設定が肥大化するため）、という理由もあります。

:::


# 5. その他のパッケージマネージャーとの共存
Linux や WSL では apt などを利用するかと思います。

Nix はユーザー環境にパッケージをインストールする仕組みであるため、**OS の深い部分（システム権限やデーモン）が絡むパッケージ**との相性があまり良くありません。

:::message
システムサービス（Docker のデーモンなど）を管理しようとすると、設定ファイルの配置場所や権限の関係で、記述が複雑になります。
:::

また、GUI アプリも管理が困難になりがちです。

:::message
GUI アプリの中には、設定ファイルや依存ライブラリが必ず /usr/lib にあるといった前提で作られているものがあり、全てのファイルを /nix/store という特殊な場所に隔離する Nix とは根本的に相性が悪いです。

WSL の場合は Windows ホスト側も処理に関わってくるので、かなり厳しいです。
:::

そのため、システム権限・デーモンが関連するパッケージや GUI アプリは apt などで管理した方が無難かと思います。
