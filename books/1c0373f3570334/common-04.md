# 1. この章でやること
この章は「[Homebrew](https://brew.sh/ja/) や [mise](https://github.com/jdx/mise) と競合しないか不安な人向け」の解説です。
Nix と各種パッケージマネージャーをどう使い分けるかについても紹介します。


# 2. 競合と共存に向けた考え方
同じパッケージが複数の仕組みからインストールされた場合、**どれを使うかは PATH（実行ファイルの探索順）で決まります**。

Nix・Homebrew・mise のいずれもパッケージのインストール先（バイナリの管理方法）が異なるため、**共存可能**です。
しかし、同じパッケージを複数で管理すると、どれが使われているか不明瞭になりがちです。

そのため、**各マネージャーの役割分担を明確にしておくことを推奨します**。
**また、Nix にも強み・弱みがあるため、適度に他のパッケージマネージャーと併用すべきです**。


:::details どのパッケージマネージャー管理下であるかを確認する方法
以下のコマンドで確認できます。

```zsh:Zsh
# どのパスのツールが使われているか確認
which git

# 重複してインストールされているか確認
which -a git
```

以下の例では Nix 管理下の Git が利用されており、他に Homebrew とデフォルトでインストールされている Git もあると分かります。

```zsh:Zsh
> which git
/etc/profiles/per-user/ryu/bin/git

> which -a git    
/opt/homebrew/bin/git
/etc/profiles/per-user/ryu/bin/git
/usr/bin/git
```

:::


# 3. Homebrew と Nix の共存
[Homebrew](https://brew.sh/) は Mac 向けの GUI アプリや CLI ツールが豊富なパッケージマネージャーです。

Nix も Mac 専用のパッケージを扱うことができますが、マイナーなパッケージだと登録されていない場合もあります。
**そのため、Mac においては Homebrew と併用するのが現実的と言えます**。

次の方針にするとシンプルになるかと思います。

- Nix で一般的なパッケージを揃える（開発に利用するツールなら大抵は揃っています）
- Nix にないパッケージのみ Homebrew で揃える

この分担にしておけば、Homebrew の強みを活かしつつ、Nix の利点（再現性）を享受できます。


# 4. mise と Nix の共存
[mise](https://mise.jdx.dev/) は言語ランタイムや CLI ツールの管理を手軽に行えるのが強みです。
パッケージ管理の他に、タスクランナーや環境変数の設定機能を利用している方も多いかもしれません。

**Nix でも同様のことができるため、役割が重なりやすいです**。
うまく両者を併用しつつ、**役割を明確に分ける**ことを推奨します。

- 例 1
  - 言語ランタイムの管理は mise
  - それ以外のパッケージの管理は Nix
  - 環境変数やタスクの管理は mise
- 例 2
  - パッケージの管理は Nix
  - 環境変数やタスクの管理は mise

:::message
**Node.js、Python、Terraform といったバージョンを指定・切り替えしたいツールの場合、Nix では実現が手間です（書き方を学ぶ必要があります）**。

**ですので、これらを mise で管理するのはアリだと思います**。

詳細は Tips（[個別にパッケージのバージョンを管理する方法](https://zenn.dev/trifolium/books/1c0373f3570334/viewer/tips-03)）で解説しています。

----

また、タスクランナーや環境変数の管理といった機能も、mise やその他のツールで管理するのが楽だと思います。

**Nix で実現する場合、独自の書き方（正確には、Flakes の devShell という機能）を学ぶ必要があり、敷居が高くなります**。

**個人的には、Nix はパッケージの管理に専念させ、それ以外は他のツールで補うのをおすすめします**。

>※筆者個人の好みとして、1 つのツールに責務を過剰に持たせることを好まない（ツールへの依存が強まりツール変更が困難になる、設定が肥大化する）、という理由もあります。

:::


# 5. その他のパッケージマネージャーとの共存
Linux や WSL では apt などを利用するかと思います。

通常、Nix はユーザー環境にパッケージをインストールする仕組みであるため、**OS の深い部分（システム権限やデーモン）が絡むパッケージ**との相性があまり良くありません。

:::message
**例えば、システムサービス（Docker のデーモンなど）を管理しようとすると、設定ファイルの配置場所や権限の関係で、記述が複雑になります**。
:::

また、GUI アプリも管理が困難になりがちです。

:::message
一部の GUI アプリでは、設定ファイルや依存ライブラリが特定の場所（`/usr/lib` など）にある前提で作られているものがあり、パッケージを `/nix/store` に隔離する Nix では相性悪く、設定が煩雑になります。

また、**WSL の場合は Windows ホスト側も処理に関わってくるので、より顕著です**。
:::

そのため、**システム権限・デーモンが関連するパッケージは apt などで管理した方が無難**かと思います。
**GUI アプリの場合は、Nix で導入を試し、うまく動かなければ apt に管理を移すのが楽です**。

>有名なアプリなら対処法があるかもしれませんが、Nix は英語文献がメインなので調査難度が高いです。

:::message
Nix でシステム領域まで管理したい場合、[system-manager](https://github.com/numtide/system-manager) を利用する、Mac なら [nix-darwin](https://github.com/nix-darwin/nix-darwin) を使う、NixOS というディストリビューションを使う、といったアプローチがあります。

Nix 沼にハマったら検討してみても良いかもしれません。
:::
