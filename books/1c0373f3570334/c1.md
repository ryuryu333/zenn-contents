# 1. はじめに
この本では、いきなりすべてを Nix 化するのではなく、**小さく始めて、徐々に置き換える**ことをテーマとしています。

この本の読者は、すでに Homebrew などで環境構築を行っているユーザーを想定しています。

具体的な導入方法や技術的解説の前に、本章では「**Nix を入れると最終的にどんな事ができるのか？**」を紹介します。

:::message
理解しやすい様に簡易化したコード例も記載します。
詳細は次章以降で解説します。
ここでは、本書の内容を全て実施した後の雰囲気（＝**Nix で設定を宣言的に記述する**）を知っていただければ幸いです。
:::

:::message
**本書は主に Mac ユーザー を対象として解説します**。

基本的な仕組みは Linux (WSL2 含む) でも共通ですが、Mac 特有の設定（nix-darwin）を含んでいます。
Windows/Linux ユーザーの方は nix-darwin と書かれた箇所は読み飛ばしてください。
:::


# 2. ユーザー環境の構築
## 2.1 モチベーション
Windows11（WSL）と MacBook をプライベートでは利用しています。

私は面倒くさがりなので、マシン間でいかに手軽に環境を再現できるかを重視しています。
また、ソフトのバージョン違いに起因するエラーを考慮したくないので、依存を含む全てのソフトのバージョンを再現したいと考えています。

**そこで、採用したのが Nix です**。

現在は、一つの GitHub リポジトリに dotfiles（`.gitconfig` 等）を集約し、かつ、ソフトの導入・設定を管理しています。
Nix のロックファイルを git 管理し、依存を含むソフト一式のバージョンを統一・再現できる様にしています。


## 2.2 環境を移行する場合
仮に、MacBook を買い替えた場合、以下の 3 ステップでユーザー環境が再現されるので、移行作業は一日もかかりません。

- Nix をインストール
- git clone <my_dotfiles>
- ビルドコマンドを実行する


## 2.3 ツールの追加
ツールの追加作業もシンプルです。

```bash:Bash
brew install git
```

する代わりに、以下の様に設定ファイル（`home.nix`）にツール名を追記し、`sudo darwin-rebuild switch` を実行するだけです。

```nix:dotfiles/home-manager/home.nix
home.packages = with pkgs; [
    ...
    git  # git をインストール
];

home.file = {
    # 設定ファイルを配置
    ".gitconfig".source = ../git/.gitconfig;
};
```

```bash:Bash
sudo darwin-rebuild switch
```

>※WSL/Linux の場合は `home-manager switch`

`git push` しておけば、別の PC でも同じバージョン・同じ設定の git を利用できます。


# 3. 開発環境の構築
## 3.1 モチベーション
環境差異に起因したエラーを考慮したくないので、私はプロジェクトの実行環境の再現性を重視しています。

過去では、プロジェクトごとに Docker コンテナ内に環境を構築していました。
しかし、ここまで極端にすると日常のちょっとした開発には運用が重いです。

かといって、プロジェクト A 用のツールがプロジェクト B から見える（影響を及ぼし得る）ことは許容できません。

そこで、Nix を採用しました。

Nix で開発に必要なツールを依存から含めてバージョン固定し、各ツールが揃った専用の環境を構築できます。
また、Nix で構築した環境内からホスト環境にアクセスできる（ユーザー環境の git 等が使える）ため、Docker よりも手軽な運用にできます。

>※より厳密な再現性を求める場合は、Docker と Nix を組み合わせることもできます。


## 3.2 開発環境の例
私が普段利用している環境例を紹介します。

プロジェクトディレクトリに設定ファイル（`flake.nix`）を作成し、以下の様に使用するツールを記述すると、各ツールがその環境内でのみ利用可能になります。

```nix:work/data_analysis/flake.nix
devShells.default = pkgs.mkShell {
  packages = with pkgs; [
    terraform
    google-cloud-sdk
    dbt
    go-task
  ];
};
```

Python を扱う場合、Python ライブラリ管理は uv に移譲しつつ、それ以外のツールを Nix で構築しています。

```nix:work/python_project/flake.nix
devShells.default = pkgs.mkShell {
  packages = with pkgs; [
    python312
    uv
    ollama
    go-task
  ];
};
```

direnv と併用すると、各ディレクトリを開くと自動的に Nix による環境を起動できます。

```bash:Bash
$ cd work/python_project

$ uv --version
uv 0.9.18

$ cd ..

$ uv --version
command not found: uv
```

```bash:Bash
$ cd work/python_project

# Nix で構築した環境内からユーザー環境のツールを利用可能
$ git --version
git version 2.51.2
```

また、プロジェクト毎に別々の Python や Node.js を入れることも容易です。
プロジェクト毎に生成されるロックファイルに基づいてツールのバージョンが固定されるので、プロジェクトによって別々のバージョンを利用できます。


# 4. 導入の進め方 - ちいさく始める
Nix を活用すると上記の様な運用が可能になりますが、Nix の様々な機能・ツールを組み合わせる必要があります。

これらを一足飛びに導入しようとすると、学習コストが膨大となり、導入のハードルが高く見えがちです。

そこで、本書では以下のように段階的に導入します。

1. home-manager でユーザー環境（CLI ツールなど）を管理する
2. home-manager で管理できないツール（Homebrew にしかないツール）を nix-darwin で補完する
3. flakes devShell で開発環境を分離・再現する
4. nix-darwin で Mac のシステム設定も管理する

この順序で進めることで、**無理なく Nix の恩恵を実感**できる構成にしています。

:::message
まずはステップ 1 を試してみて、良さそうと思ったら Nix で管理する対象を増やしていくと良いと思います。

開発という観点なら、ステップ 1 と 3 の内容だけでも十分強力です。

ステップ 2 は必須ではありませんが、開発外で利用するアプリは Nix にない事が多いので、Nix で管理したい場合は必要になります。
ステップ 3 には影響が無いので、飛ばしても大丈夫です。

ステップ 4 は発展的な内容です。Nix にハマったら試してみてください。
ステップ 1 ~ 4 全てを行うと、Mac 全体を宣言的に管理でき、本章で紹介した様に「Mac の引越し作業が一日で終わる」ので、とても楽です。
:::
