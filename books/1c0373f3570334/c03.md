# 1. この章でやること
この章は「Homebrew や mise と競合しないか不安な人向け」の補足です。
気にならなければ飛ばして先へ進んでも問題ありません。


# 2. 競合が起きる理由
同じツールが複数の仕組みからインストールされた場合、**どこからインストールされたツールを使うかは PATH（実行ファイルの探索順）で決まります**。

Nix・Homebrew・mise のいずれも**共存自体は可能**です。
しかし、同じツールを複数で管理すると、「どれが使われているかわからない」状態になりがちです。

この章では、**共存の考え方**と**最低限の確認方法**を扱います。


# 3. Homebrew と Nix の共存
Homebrew は Mac 向けの GUI アプリや CLI ツールが豊富です。
一方、Nix は Mac 専用のアプリ等を直接は導入できません。

両方を併用する場合は、次の方針で考えると混乱しにくくなります。

- CLI ツールは Nix 側で揃える（開発に利用するツールなら大抵は揃っています）
- Homebrew は GUI アプリや Mac 専用のアプリを管理する

この分担にしておけば、Homebrew の強みを活かしつつ、Nix の利点（再現性）を享受できます。


:::details どちらが使われているかを確認する
以下のコマンドで確認できます。

```bash:Bash
# どのパスのツールが使われているか確認
which git

# 重複してインストールされているか確認
which -a git
```

:::


# 4. mise と Nix の共存
mise は言語ランタイムや CLI ツールのバージョン管理を手軽に行えるのが強みです。
`.mise.toml` で特定ディレクトリでのツールバージョンを固定するのに加えて、タスクランナーや環境変数の設定機能を利用している方も多いかもしれません。

Nix でも同様のことができるため、**役割が重なりやすい**です。

両方を併用する場合は、以下のように役割分担を決めておくことを推奨します。

- 例 1
    - 言語ランタイムの管理は mise
    - それ以外のツールの管理は Nix
    - 環境変数やタスクの管理は mise
- 例 2
    - ツールの管理は Nix
    - 環境変数やタスクの管理は mise

Nix に慣れてから徐々に移行していくと学習負荷を抑えられると思います。


:::message
Node.js、Python、Terraform といったバージョンを指定・切り替えしたいツールの場合、Nix では実現が手間です（書き方を学ぶ必要があります）。
ですので、これらを mise で管理するのはアリだと思います。

詳細は開発環境の構築（Nix Flakes devShells）に関する章で解説します。
:::
