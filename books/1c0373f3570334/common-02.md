# 1. パッケージマネージャー Nix とそれ以外の Nix
Nix という単語には複数の意味合いが内包されおり、慣れていない間は勘違いしやすいです。

- [Nix](https://github.com/NixOS/nix)：パッケージマネージャー
- [Nix 言語](https://nix.dev/tutorials/nix-language.html)：DSL（Domain Specific Language）、Nix の設定記述に利用
- [Nixpkgs](https://github.com/NixOS/nixpkgs)：Nix の公式パッケージリポジトリ
- [NixOS](https://nixos.org/)：Nix をベースにした Linux ディストリビューション

どれも Nix と名前が付いていますが別物です、本書では上記リストの通りに呼び分けます。
**本書では、パッケージマネージャー Nix について解説していきます**。


# 2. Nix の特徴
[公式サイト](https://nixos.org/)では Reproducible（再現性）、Declarative（宣言的）、Reliable（信頼性）が特徴として提示されています。


## 2.1 再現性
Nix はパッケージのビルドを個々に独立して行います。
ビルドに必要な情報が全て宣言的に管理されるため、**異なる PC でも依存含むパッケージを再現できます**。


## 2.2 宣言的
**Nix では環境の設定は全てテキストで宣言します**。
設定ファイル（`*.nix`）があれば同じ環境を再現できるため、Git 管理と相性が良いです。


:::details 命令的 vs 宣言的
以下に示すように、Nix は一般的なパッケージマネージャーとアプローチが異なります。
**アプローチが違うが故に、操作方法・設定の書き方も変わってきます**。

----

#### 命令的（Imperative）
従来の多くのパッケージマネージャー（apt、brew、pip など）での操作です。
「A をインストールする」「B をアップデートする」といったコマンド（命令）を積み重ねて環境を構築します。

```zsh:Zsh
brew install git
```

**コマンドを実行した順番や時期によって、最終的な環境の状態が異なる（＝再現性が低くなる）可能性があります**。

>無論、`brew bundle` と `Brewfile.lock.json` のように、パッケージ一覧とロックファイルを作成することで、再現性を確保する仕組みが搭載されていることも多いです。
しかし、ベースとなる仕組みだけでは再現性が低く、追加の操作が必要となります。


#### 宣言的（Declarative）
Nix が採用しているアプローチです。
「あるべき状態」をファイルに記述します。

```nix:設定例
home.packages = with pkgs; [
  git
];
```

```zsh:Zsh
# 設定を反映
home-manager switch
```

**`brew install` を禁止して `brew bundle` と `Brewfile.lock.json` のみでパッケージを追加・削除するようなイメージです**。

>他の例をあげると、「手順」ではなく「状態」を記述する IaC（Infrastructure as Code）の考え方と近いかもしれません。
:::


## 2.3 信頼性
パッケージ A をアップデートしたら、パッケージ B が依存しているライブラリ X が勝手に更新された、といったことを仕組みで防ぎます。
これにより、**[依存関係地獄](https://ja.wikipedia.org/wiki/%E4%BE%9D%E5%AD%98%E9%96%A2%E4%BF%82%E5%9C%B0%E7%8D%84)を解消しています**。


:::details Nix と依存関係地獄
一般的なパッケージ管理を例に挙げます。

```:ありがちな挙動
パッケージ A をインストール
-> ライブラリ X-1.1 が依存として必要
-> X-1.1 をインストール

パッケージ B をインストール
-> ライブラリ X-2.0 以上が依存として必要
-> X-1.1 が既にある
-> X を 2.0 にアップデート
```

**この場合、パッケージ `A` が `X-1.1` の挙動に強く依存し、`X-2.0` に対応していなかった場合、パッケージ `A` の動作が意図せず変わる（＝いわゆる壊れる）可能性があります**。

----

**一般的なパッケージマネージャーと Nix の分かりやすい違いはバイナリの保管場所です**。

```バイナリの保管場所
# 一般的なパッケージマネージャー
usr/lib/X  # A と B が依存として利用

# Nix
nix/store/abcdefg-X-1.1  # A が利用
nix/store/hijklmn-X-2.0  # B が利用
```

**Nix では同じパッケージであっても異なるバージョンを同時に保持することが出来ます**。

これにより、依存関係地獄を回避しています。

----

Nix では以下のようなパスにパッケージを保管します。

```バイナリの保管場所
nix/store/<ハッシュ値>-<パッケージ名>-<バージョン>
```

**ハッシュ値はビルドに使う入力から計算されます**。

同じ `X-1.1` であっても、依存となるライブラリのバージョン等が異なる場合（ビルドに使う入力が異なる場合）、別々のパッケージとして扱われます。

**そのため、Nix では `nix/store/abcdefg-X-1.1`、`nix/store/opqrstu-X-1.1` のように同一バージョンが同時に存在することも起こり得ます**。
:::


# 3. Nix 言語について
本書では深掘りしませんが、Nix は JSON や YAML ではなく、**純粋関数型言語である Nix 言語**を利用します。

先ほど、Nix はパッケージのビルドに関わるインプットからハッシュ値を計算し、パッケージを区別していると紹介しました。
**常に同じ入力に対して同じ出力を返す性質**を言語に持たせることで、この特徴を実現していると言えます。


:::message
**関数型言語というと難しそうに思えますが、実用上ユーザーが編集する箇所はシンプルです**。

例えば、特定の開発環境内で利用するパッケージを宣言する場合、以下のようにリスト `[]` にパッケージ名を列挙するだけです。

```nix:設定例
packages = with pkgs; [
  python312
  uv
  go-task
];
```

**Nix の設定ファイルは使いまわせるコードが多いので、テンプレートを複製し、環境によってリストの中身を書き換える、といった運用ができます**。
:::


また、Nix 言語は**変数や if などのロジックを利用できる**ため、記述の柔軟性が高いです。
一般的なプログラミング言語の感覚で処理を抽象化したりできるので、慣れると可読性・拡張性を意識したコードを書きやすく便利です。

>ちょっと複雑な YAML だと思えば大丈夫です。
