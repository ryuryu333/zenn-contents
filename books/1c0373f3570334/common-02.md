# 1. Nix とは？
Nix という単語には複数の意味合いが内包されおり、慣れていない間は勘違いしやすいです。
まず、最初に一覧を提示します。

- [Nix](https://github.com/NixOS/nix)：パッケージマネージャー
- [Nix 言語](https://nix.dev/tutorials/nix-language.html)：DSL（Domain Specific Language）、Nix の設定記述に利用
- [Nixpkgs](https://github.com/NixOS/nixpkgs)：Nix の公式パッケージリポジトリ
- [NixOS](https://nixos.org/)：Nix をベースにした Linux ディストリビューション

**本書では、パッケージマネージャー Nix について解説していきます**。


# 2. Nix の特徴
Nix は「再現性の高さ」と「宣言的記述」が特徴のパッケージマネージャーです。


### 2.1 再現性
Nix はパッケージのビルドを個々に独立して行います。
また、ビルドに関する入力に基づきハッシュ値を計算し、パッケージのプレフィックスとして利用しています。

そのため、**同じパッケージ・同じバージョンであっても、入力（依存ライブラリのバージョンなど）が異なる場合、別のパッケージ扱い**されます。

**この特性により、[依存関係地獄](https://ja.wikipedia.org/wiki/%E4%BE%9D%E5%AD%98%E9%96%A2%E4%BF%82%E5%9C%B0%E7%8D%84)を解消しています**。

:::details Nix と依存関係地獄
一般的なパッケージ管理を例に挙げます。

パッケージ `A` をインストールした際、依存関係の解消のために**暗黙的に**ライブラリ `X-1.1` がインストールされたとします。
次に、パッケージ `B` をインストールします。このパッケージは `X-2.0` を必要としていました。
この際、パッケージマネージャーは既存の `X-1.1` を `X-2.0` にアップデートする、という挙動をとることが多いでしょう。

この場合、パッケージ `A` が `X-1.1` の挙動に強く依存し、`X-2.0` に対応していなかった場合、パッケージ `A` の動作が意図せず変わる（＝いわゆる壊れる）可能性があります。

----

通常のパッケージマネージャーでは、`usr/lib/X` のように 1 つのディレクトリ内でパッケージを管理します。
**一方、Nix では、`nix/store/abcdefg-X-1.1`、`nix/store/hijklmn-X-2.0` のように独立したパッケージとして管理します**。

これにより、依存関係地獄を回避しています。

>nix/store/<ハッシュ値>-<パッケージ名>-<バージョン> という並びです。
>同じ `X-1.1` であっても、依存となるライブラリのバージョン等が異なる場合、別々のパッケージとして扱われます。
つまり、`nix/store/abcdefg-X-1.1`、`nix/store/opqrstu-X-1.1` が同時に存在することも Nix では起こり得ます。

:::


### 2.2 宣言的
Nix では、必要なパッケージをコードとして記述します。
「**最終的にどのような状態になっていて欲しいか**」を設定ファイルに**宣言する**アプローチです。

>「手順」ではなく「状態」を記述する IaC（Infrastructure as Code）の考え方と近いです。

テキストベースであるため Git 管理と相性が良いです。
パッケージの追加・除外のトラッキング、複数マシン間での共有・再現、といったことが容易になります。


:::details 命令的 vs 宣言的
以下に示すように、Nix は一般的なパッケージマネージャーとアプローチが異なります。
**アプローチが違うが故に、操作方法・設定の書き方も変わってきます**。

#### 命令的（Imperative）
従来の多くのパッケージマネージャー（apt、brew、pip など）での操作です。
「A をインストールする」「B をアップデートする」といったコマンド（命令）を積み重ねて環境を構築します。
コマンドを実行した順番や時期によって、最終的な環境の状態が異なる（＝再現性が低くなる）可能性があります。

>無論、`brew bundle` と `Brewfile.lock.json` のように、ツールの一覧とロックファイルを作成することで、再現性を確保する仕組みが搭載されていることも多いです。
しかし、ベースとなる仕組みだけでは再現性が低く、追加の操作が必要となります。


#### 宣言的（Declarative）
Nix が採用しているアプローチです。
「パッケージ A と B がインストールされている状態であるべき」とファイルに記述します。
Nix はそのファイルを読み込み、足りないものをインストールし、不要なものを削除して、記述通りの状態（あるべき姿）に収束させます。

これにより、Nix は高い再現性・可搬性を確保しています。
:::


# 3. Nix 言語について
本書では深掘りしませんが、先述の「再現性」と「宣言的記述」を実現するために、Nix は JSON や YAML ではなく、**純粋関数型言語である Nix 言語**を利用します。

例えば、先ほど、Nix はパッケージのビルドに関わるインプットからハッシュ値を計算し、パッケージを区別していると紹介しました。
**常に同じ入力に対して同じ出力を返す性質**を言語に持たせることで、この特徴を実現していると言えます。

----

**関数型言語というと難しそうに思えますが、実用上ユーザーが編集する箇所はシンプルです**。

例えば、環境内で利用するパッケージを宣言する場合、以下のようにリスト `[]` にパッケージ名を列挙するだけです。

```nix:設定例
packages = [
  pkgs.git
  pkgs.python312
  pkgs.uv
];
```

リスト以外は基本的に使いまわせるコードとなるので、テンプレートを複製して使い回し、環境によってリストの中身を書き換える、といった運用をします。

----

また、Nix 言語は**変数や if などのロジックを利用できる**ため、記述の柔軟性が高いです。
一般的なプログラミング言語の感覚で処理を抽象化したりできるので、慣れると可読性・拡張性を意識したコードを書きやすく便利です。

>ちょっと複雑な YAML だと思えば大丈夫です。
