## 1. Nix を利用する方法
### 1.1 前置き
この記事の環境を前提としています。

https://zenn.dev/trifolium/articles/007bff63247432

:::message

- 環境の概略
  - Nix 導入済み
  - `flake.nix`、`devShell` を利用

:::

**`importNpmLock` を用いて、Node パッケージを Nix の `devShell` で利用可能にします**。

詳細はこちらの記事で解説しています。

https://zenn.dev/trifolium/articles/6678b0c0fb0d27


### 1.2 Node パッケージ管理用のファイルを作成
プロジェクト直下に `node-pkgs` フォルダを作成し、`package.json` を作成します。

```diff:フォルダ構成
  zenn_contents/
+ ├─ node-pkgs/
+ │  └─ package.json
  └─ flake.nix
```

```json:package.json
{
  "name": "zenn-cli-env",
  "version": "1.0.0",
  "private": "true",
  "devDependencies": {
    "cspell": "9.2.1",
    "markdownlint-cli2": "0.18.1",
    "textlint": "15.2.2",
    "textlint-filter-rule-comments": "1.2.2",
    "textlint-rule-preset-ja-spacing": "2.4.3",
    "textlint-rule-preset-ja-technical-writing": "12.0.2",
    "textlint-rule-terminology": "5.2.15",
    "zenn-cli": "0.2.3"
  }
}
```

:::message
パッケージのバージョンが古いかもしれませんが、以降の工程で更新します。
:::


### 1.3 flake.nix の作成

```nix:flake.nix
{
  description = "Zenn CLI environment";

  inputs = {
    nixpkgs.url = "github:nixos/nixpkgs/nixpkgs-unstable";
    flake-utils.url = "github:numtide/flake-utils";
  };

  outputs =
    { nixpkgs, flake-utils, ... }:
    flake-utils.lib.eachDefaultSystem (
      system:
      let
        pkgs = nixpkgs.legacyPackages.${system};
        inherit (pkgs) importNpmLock;
        nodejs = pkgs.nodejs_24;
        npmRoot = ./node-pkgs;
      in
      {
        devShells.default = pkgs.mkShell {
          packages = [
            pkgs.treefmt
            pkgs.lychee
            importNpmLock.hooks.linkNodeModulesHook
          ];
          npmDeps = importNpmLock.buildNodeModules {
            inherit npmRoot nodejs;
          };
        };

        # for updating package.json and package-lock.json
        devShells.node = pkgs.mkShell {
          packages = [
            nodejs
            pkgs.npm-check-updates
          ];
        };
      }
    );
}
```

:::message alert
この時点だと、**`nix-direnv` で `nix develop` を自動化している場合、ビルドエラーが発生します**。

原因は、`importNpmLock.buildNodeModules` の処理にあります。
この機能は `package-lock.json` を必要とします。
現時点では必要なロックファイルが無いので、エラーになります。

**次の工程でロックファイルを用意するまでの間、一時的に `nix-direnv` を無効化してください**。
:::


### 1.4 パッケージの更新
先ほど `package.json` に記述したパッケージのバージョンが古いかもしれないので、最新へ更新します。

`node-pkgs` に移動します。

```bash
cd your_path/zenn_contents/node-pkgs
```

`devShells.node` 環境を利用して、`ncu` コマンドを実行します。

```bash
nix develop .#node -c ncu -u
```

:::message
`package.json` に古いパッケージがあった場合、最新バージョンに書き換わります。
:::

https://www.npmjs.com/package/npm-check-updates


### 1.5 package-lock.json の作成
`npm` を利用して `package.json` に記載された Node パッケージの依存関係を `package-lock.json` に記述します。

```bash
nix develop .#node -c npm install --package-lock-only
```

### 1.6 devShell の起動
プロジェクト直下に戻り、`nix develop` で環境を起動します。
`package-lock.json` に基づいてパッケージが構築され、プロジェクト直下に `node_modules` がリンクされます。

```bash
cd ..
```

```bash
nix develop
```

**これで準備は完了です**。

:::message
**この一連の作業、面倒だしコマンドが複雑で間違えそう、と感じたでしょうか**。

**`Taskfile` というツールを用いると、簡略化できます**。
（**e.g. `task add -- zenn-cli`、`task update`**）

**別の記事で解説する予定です**。
:::

# 各リンターの設定
## markdownlint-cli2
`linter` ファルダと `.markdownlint-cli2.jsonc` を作成します。

```diff:フォルダ構成
  zenn_contents/
+ └─ linter/
+     └─ .markdownlint-cli2.jsonc
```

```json:.markdownlint-cli2.jsonc
{
    "config": {
        "MD001": false,
        "MD012": false,
        "MD013": false,
        "MD022": false,
        "MD024": false,
        "MD025": false,
        "MD029": false,
        "MD033": false,
        "MD034": false 
    }
}
```

ルール一覧（`MDxxx`）はこちらのページに記載されています。

https://github.com/DavidAnson/markdownlint

### 無効化したルールと理由
#### MD001 見出しレベルは一度に 1 レベルずつ増加します

```markdown:NG
# Heading 1

### Heading 3
```

```markdown
# 見出し
#### 目次に表示したくない見出し
こう書きたい時がある。
```


#### MD012: 連続する複数の空白行

```markdown:NG
text


text
```

```markdown
# 見出し
解説


# 次の見出し
こう書けた方が編集中に読みやすい。
個人の好み。
```


#### MD013: 行の長さ
80 文字を超えると警告が発生する。
`textlint` と役割が重複しているので、無効化。


#### MD022: 見出しは空白行で囲む必要があります

```markdown:NG
# Heading 1
Some text

Some more text
## Heading 2
```

```markdown
# 見出し
こう書けた方が読みやすい。
個人の好み。
```


#### MD024: 同じ内容の複数の見出し

```markdown:NG
# Some text

## Some text
```

```markdown
# 紹介
## textlint

# 使い方
## textlint

こういった書き方をしたいから。
```


#### MD025: 同じ文書内に複数のトップレベル見出しがある

```markdown:NG
# Top level heading

# Another top-level heading
```

```markdown
# はじめに

# 概要

# おわりに

こういった書き方をしたいから。
```


#### MD029: 順序付きリスト項目の接頭辞

```markdown:NG
- 1. まずこれをする
- 2. 次のそれをする
```

```markdown
以下の様な書き方を強制するルールです。

1. Do this.
2. Do that.
3. Done.

これだけなら無害だったのですが、NG 例の書き方でも警告が出ました。

NG の様な書き方は使うので無効化。
```


#### MD033: インライン HTML

```markdown:NG
<h1>Inline HTML heading</h1>
```


```markdown
|||
|--|--|
|改行無し|改行<br>有り|

表の中で改行する際に HTML タグを利用するから。
```


#### MD034: 裸の URL が使用される

```markdown:NG
For more info, visit https://www.example.com/ or email user@example.com.
```

```markdown
https://zenn.dev/zenn/articles/markdown-guide

Zenn 独自の記法にリンクカードがあり、使いたいから。
```


## textlint
`.textlintrc.json` を作成します。

```diff:フォルダ構成
  zenn_contents/
  └─ linter/
      ├─ .markdownlint-cli2.jsonc
+     └─ .textlintrc.json
```

```json:.textlintrc.json
{
  "plugins": {},
  "filters": {
    "comments": true
  },
  "rules": {
    "preset-ja-spacing": {
      "ja-space-between-half-and-full-width": {
        "space": "always"
      }
    },
    "preset-ja-technical-writing": {
      "ja-no-weak-phrase": false,
      "no-mix-dearu-desumasu": {
        "preferInList": "ですます"
      },
      "ja-no-mixed-period": {
            "allowPeriodMarks": [":"]
      },
      "no-exclamation-question-mark": false
    },
    "terminology": {
      "exclude": [
        "V[ -]?S[ -]?Code"
      ]
    }
  }
}
```

### 追加した拡張
#### textlint-filter-rule-comments
`textlint` でチェックしない領域を指定できます。

```markdown:markdown
<!-- textlint-disable -->

This is ignored text by rule.
Disables all rules between comments

<!-- textlint-enable -->
```

`.textlintrc.json` では、この拡張を有効化するだけでカスタマイズはしていません。

```json:.textlintrc.json
{
  "filters": {
    "comments": true
  }
}
```


#### textlint-rule-preset-ja-spacing
日本語周りにおけるスペースの有無を決定する `textlint` ルールプリセットです。

デフォルトでは、半角・全角の間にスペースを**入れない**設定になっています。

私の好みで「英単語や半角数字」と日本語はスペースを入れたいので、**入れる**設定に変更しています。

```json:.textlintrc.json
{
  "rules": {
    "preset-ja-spacing": {
      "ja-space-between-half-and-full-width": {
        "space": "always"
      }
    }
  }
}
```


#### textlint-rule-preset-ja-technical-writing
技術文書向けの `textlint` ルールプリセットです。

下記をカスタマイズしています。

- ja-no-weak-phrase
`〜と思います`といった弱い表現を禁止します。
そういった表現を使いたいので、許可に変更しています。

- no-mix-dearu-desumasu
見出しは自動、本文はですます調、箇条書きはである調、で統一するルール。
本文・箇条書きどちらも「ですます調」に統一するように変更しています。

- ja-no-mixed-period
「。」のつけ忘れチェックを行うルール。
例外として許可したい文字列（「。」として認識する文字列）として「:」を追加しています。
追加しないと、以下の構文で警告が出ます。

```markdown:markdown
:::message
メッセージをここに。
:::
```

- no-exclamation-question-mark
感嘆符!！、疑問符?？を禁止するルール。
使えた方が便利なので許可に変更しています。


```json:.textlintrc.json
{
  "rules": {
    "preset-ja-technical-writing": {
      "ja-no-weak-phrase": false,
      "no-mix-dearu-desumasu": {
        "preferInList": "ですます"
      },
      "ja-no-mixed-period": {
            "allowPeriodMarks": [":"]
      },
      "no-exclamation-question-mark": false
    }
  }
}
```


#### textlint-rule-terminology
英語の技術文書内の用語、ブランド、テクノロジーのスペルをチェックして修正するためのルールです。

```markdown:NG例
Javascript → JavaScript
NPM → npm
front-end → frontend
website → site
Internet → internet
```

**この拡張は継続的な設定の見直しが必要だと思います**。

例えば、通常だと VSCode はエラーになります。
（Visual Studio Code 表記にすべき）

![textlint のエラー](/images/5b01a68b80808b/5b01a68b80808b-2025-9-5.webp)

**このエラーを止めたいと仮定して、方法を解説します**。

まず、公式の GitHub リポジトリで辞書を確認します。

https://github.com/sapegin/textlint-rule-terminology/blob/master/terms.jsonc

`Visual Studio Code` で検索すると、以下が見つかります。

```json:terms.jsonc
  ["Visual ?Studio ?Code", "Visual Studio Code"],
  ["V[ -]?S[ -]?Code", "Visual Studio Code"],
```

`.textlintrc.json` の `rules.terminology.exclude` に `V[ -]?S[ -]?Code` を追加します。

これで VSCode は許容されます。
なお、`Visual ?Studio ?Code` は除外指定していないので、visual studio code はエラーが出ます。


```json:.textlintrc.json
{
  "rules": {
    "terminology": {
      "exclude": [
        "V[ -]?S[ -]?Code"
      ]
    }
  }
}
```



## cspell
`linter/cspell.json` に対象と辞書を記載しています。

```json:linter/cspell.json
{
  "version": "0.2",
  "files": ["articles/*.md"],
  "words": ["treefmt", "Zenn"]
}
```

- 固有名詞やツール名は辞書に追加し、誤検知を軽減
- ファイル対象は `articles/*.md` に限定


## lychee（リンク切れチェック）
- オプション: `--timeout 10`、`--max-retries 2`
  - フレーク対策としてタイムアウト短め＋少回数のリトライ
- 一時的に落ちているサイトでアラートが出た場合は、再実行や除外リストの導入を検討


# 運用のポイント
- 新規記事から順次適用し、既存記事は `excludes` を外しながら段階的に整備
- どうしても引っかかる箇所は、ルールの根拠を見直すか、ピンポイントで無効化
- 実行は `task` に集約し、手元で同じコマンドを呼ぶ

## `excludes` の外し方（段階移行）
`linter/treefmt.toml` の `[global].excludes` に既存記事を列挙しています。対応した記事から順にエントリを削除し、`task` で通ることを確認します。

```toml:linter/treefmt.toml（抜粋）
[global]
excludes = [
  "articles/00341ed49b7935.md",
  # ... 対応できたら削除
]
```

## 辞書・除外の更新
- cspell の辞書は `linter/cspell.json` の `words` に追記します（固有名詞やツール名など）。
- textlint は一時的に無視したい箇所にコメントを使えます。

```markdown
<!-- textlint-disable -->
この段落は一時的に除外します。
<!-- textlint-enable -->
```

必要に応じて、ルール側の設定（`.markdownlint-cli2.jsonc` / `.textlintrc.json`）を見直して恒久対応します。




# treefmt の設定
`linter/treefmt.toml` に各リンターを登録し、実行順と対象ファイルを定義しています。

```toml:linter/treefmt.toml
[formatter.md]
command = "markdownlint-cli2"
options = ["--config", "linter/.markdownlint-cli2.jsonc"]
includes = ["articles/*.md"]
priority = 1

[formatter.text]
command = "textlint"
options = ["--config", "linter/.textlintrc.json"]
includes = ["articles/*.md"]
priority = 2

[formatter.url]
command = "lychee"
options = ["--timeout", "10", "--max-retries", "2"]
includes = ["articles/*.md"]
priority = 3

[formatter.spell]
command = "cspell"
includes = ["articles/*.md"]
priority = 4

[global]
excludes = [
  # 既存の記事は段階的に対応するため一時的に除外
]
```

- 実行順序は「体裁 → 文体 → リンク → スペル」の順に設定しました。
- 既存記事はスタイルが固まるまで `excludes` で除外し、新規記事から適用しています。



# おわりに
`treefmt` をハブにすることで、Markdown・日本語・リンク・スペルのチェックを一本化できました。設定をリポジトリに同梱しておくと、執筆端末が変わっても同じ品質基準で書き進められます。必要に応じてルールの粒度を調整し、運用しやすい形に育てていきます。
