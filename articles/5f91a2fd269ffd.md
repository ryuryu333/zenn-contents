# はじめに
プログラミングでデバッグを行う際、スタックトレース（関数の呼び出し履歴）は非常に重要な情報です。

エラーが起きたときに**どの関数からどの関数を経由してここに至ったのか**が分かれば、原因を突き止めやすくなります。

しかし、VBAでは**実行中にスタックトレースを取得する標準的な方法がありません**。

そのため、**VBA でスタックトレースを含んだログ出力を簡単に行える**ロガーアドインを作成しました。

# VBA におけるスタックトレースの課題
## スタックトレースとは
**スタックトレース**は、プログラムの**実行経路（どの関数からどの関数が呼ばれたか）**を示す情報です。

例えば、C# では `Environment.StackTrace` を用いると以下の様な表示がされます。

```:下側の関数は上側の関数から呼び出されたと読み解きます
at System.Environment.GetStackTrace(Exception e)
at System.Environment.GetStackTrace(Exception e)
at System.Environment.get_StackTrace()
at Sample.Main()
```

https://learn.microsoft.com/ja-jp/dotnet/api/system.environment.stacktrace?view=net-9.0



VBA ではこの機能が**存在しない**ため、**エラー原因の特定が困難**になります。

実際、幾つかの記事で **VBA の問題点としてスタックトレースが貧弱でありエラー解決の手間が増大している**点が挙げられています。

https://speakerdeck.com/narazaka/vbanameteta

https://qiita.com/zakuroishikuro/items/81ddd7a33fb346aeb6a0

https://darumaexcel.uijin.com/aboutooa/vba_limit.html

## VBA 標準機能の3つの限界
### 限界1：実行中のコールスタック取得不可
他言語との比較

- C#：`Environment.StackTrace`
- Python：`traceback.print_stack()`
- VBA：**機能なし**

https://learn.microsoft.com/ja-jp/dotnet/api/system.environment.stacktrace?view=net-9.0

https://qiita.com/podhmo/items/6657ae7b6ffbadc8f773


### 限界2：エラー発生時の情報不足
VBAで取得できる情報は主に以下の3つです。

- Err.Number：エラー番号
- Err.Description：エラーメッセージ
- エラー発生箇所のハイライト（GUI 上のみ）

**最も重要な「どの経路でここに来たか」が分かりません。**


### 限界3：デバッグ時の制約
VBEの呼び出し履歴ウィンドウは存在しますが、**制約があります**。

- 実行停止時のみ利用可能
- ログとして記録不可
- 顧客環境では使用不可
- 事前のブレークポイント設定が必要

以下の記事では VBE でのスタックトレース情報の確認方法を紹介されています。

https://qiita.com/Kamo123/items/e4c39f25f44876cf9e2f

# 既存解決手法
VBAでスタックトレース相当の情報を得るため、これまで様々な手法やツールが開発・活用されてきました。

本項では、5 つの手法の**特徴**を紹介します。

## 1. 手作業ログ埋め込み
### 概要
各プロシージャに `Debug.Print` を開発者が手動で挿入する最も原始的かつ直接的な方法です。

```vba
Sub MainProc()
    Debug.Print "MainProc 開始"
    ' 処理内容
    Debug.Print "MainProc 終了"
End Sub
```

### 特徴
- ✅ 利点
    - VBA 標準機能のみで実現可能
    - 導入コスト・学習コストがゼロ
    - 出力内容を完全にコントロール可能
- ❌ 欠点
    - 関数追加時に手動でログコード追加が必要
    - ヒューマンエラー（書き忘れ・記述ミス）が頻発
    - 大規模プロジェクトでは膨大な手作業が必要

### 適用場面
- ✅ 小規模な一次的コード
- ❌ 大規模プロジェクト・チーム開発

## 2. エラー伝搬による擬似スタックトレース
### 概要
エラー発生時に各プロシージャで自身の情報を追加して上位に伝搬させる方法です。

https://stackoverflow.com/questions/127645/is-it-possible-to-retrieve-the-call-stack-programmatically-in-vb6

```vba
Sub MainProc()
    On Error GoTo ErrorHandler
    SubProc
    Exit Sub
ErrorHandler:
    Err.Description = "MainProc > " & Err.Description
    Debug.Print Err.Description
    Err.Raise Err.Number
End Sub

Private Sub SubProc()
    On Error GoTo ErrorHandler
    ' エラーが発生する処理
    Dim myValue As Integer
    myValue = 1 / 0
    Exit Sub
ErrorHandler:
    Err.Description = "SubProc > " & Err.Description
    Err.Raise Err.Number
End Sub
```

```:実行結果
MainProc > SubProc > 0 で除算しました。
```

:::details エラーの伝播方法の技術的な補足
`On Error GoTo ErrorHandler`
これにより、エラーが発生した際に、`ErrorHandler` が実行されます。

`ErrorHandler:`
このコード以下に書かれた内容が、エラー時に実行されます。

`Err`
このオブジェクトにアクセスすることで、エラー発生時の情報を取得できます。

`ErrorHandler`、`Err` の詳細については下記公式リファレンスをご確認ください。

https://learn.microsoft.com/ja-jp/office/vba/language/reference/user-interface-help/err-object

https://learn.microsoft.com/ja-jp/office/vba/language/reference/user-interface-help/on-error-statement

:::

### 特徴
- ✅ 利点
    - VBA 標準機能のみで実現
    - エラー発生時に呼び出し経路を取得可能
    - 通常時のパフォーマンスへの影響が少ない
- ❌ 欠点
    - エラー時のみ有効（正常フローは追跡不可）
    - 全プロシージャにエラーハンドリングコードが必要
    - エラー処理コードの管理が煩雑

### 適用場面
- ✅ 小規模な一次的コード
- ❌ 正常フローのデバッグが必要な場面・大規模プロジェクト・チーム開発

`CallByName` 関数を利用し、プロシージャ呼び出しをログ出力機能を持つ共通のラッパー関数経由で実行する方法です。

```vba

```

・公式リファレンス
https://learn.microsoft.com/ja-jp/office/vba/language/reference/user-interface-help/callbyname-function
:::

**サンプルコード、及び、実行結果は以下の記事より引用しました。**

```vba

```

### 特徴

    - 通常の関数呼び出しと記述方法が異なり可読性が低下
    - 階層構造のトラックトレース情報がない

### 適用場面
- ✅ クラスベースの設計が確立している場合
- ❌ 標準モジュール中心の既存コード
- ❌ パフォーマンスが重要な処理
- ❌ 階層構造のトラックトレース情報が必要な場合

# はじめに：スタックトレースが欲しい理由と VBA の課題
プログラミングでデバッグを行う際、スタックトレース（関数の呼び出し履歴）は非常に重要な情報です。

エラーが起きたときに**どの関数からどの関数を経由してここに至ったのか**が分かれば、原因を突き止めやすくなります。

しかし、VBAでは**実行中にスタックトレースを取得する標準的な方法がありません**。
そのため、開発者は以下のような**手間のかかる方法**で対策する必要がありました。

- ブレークポイントを用意してコード実行を中断し、エディター（VBE）の GUI で確認する
- 自分で全関数にログ出力やエラー伝搬コードを仕込む
- 有償の商用アドイン `vbWatchdog` を導入する（$260、エラー時のみトレース可能）

**これは開発者にとって大きな負担であり、デバッグ作業の効率を下げる原因となります。**

本記事では、そうした VBA の課題を解決するために筆者が作成した VBA ロガーアドインを紹介します。
このアドインを使うと、**VBA でスタックトレースを含んだログ出力を簡単に行えます**。

:::message
**スタックトレース / コールスタック について**
厳密には、本ロガーはコード実行中のログ出力にコールスタック情報を含めることを可能とする、が正しいです。
本記事では、分かりやすさを優先し、一般的な用語であるスタックトレース呼びをしています。

- スタックトレース (Stack Trace)

エラー発生時の実行履歴を逆順に表示するもの。
- コールスタック (Call Stack)

関数の呼び出し情報を一時的に保管するメモリ領域。
:::

Private Const MODULE_NAME As String = "MyModule" ' モジュール名をここに書く

Sub UseCase()

    myLogger.Terminate ' ロガーの終了処理
End Sub

Sub MainProc()
    ' この1行を追加するだけで関数の開始・終了を記録できる
    Const PROC_NAME As String = "MainProc": Dim scopeGuard As Variant: Set scopeGuard = UsingTracer(MODULE_NAME, PROC_NAME)

    Dim hoge As MyClass
    Set hoge = New MyClass
    hoge.ClsMainProc
End Sub

# ロガーの特徴と仕組み
このロガーアドインの主な特徴について説明します。

## 呼び出し階層（スタックトレース）の可視化
ログ出力時にどの関数がどれを呼び出したかが直感的に分かる形式で記録します。 

`MyModule.SubProc < MyModule.MainProc`
左が呼び出された関数（現在実行中の関数）、`<` の右側が呼び出し元の関数を示します。

**ネストした呼び出しも階層構造としてログから読み取れるため、複雑な処理の流れを追跡しやすくなります。**

## 関数の開始・終了を1行でトレース、終了は自動検出でミス防止
関数の冒頭に 1 行追加するだけで、スタックトレース情報が更新されます。

- 開始：追加した行の最後にある `UsingTracer` の呼び出しにより記録される
- 終了：現在の関数が終了（`End Sub`、`Exit Sub`）すると自動的に処理が走り記録される

**開始と終了が紐づいているので「開始は書いて終了を書き忘れ、階層がぐちゃぐちゃになる」といったヒューマンエラーを回避出来ます。**

また、トレースが不要な関数であれば、トレース用の行を書かないという対応も可能です。
（for 文で同じ関数を繰り返し呼び出す場合など）

```vba
Private Sub SubProc()
    ' この1行で SubProc の開始ログが記録される
    Const PROC_NAME As String = "SubProc": Dim scopeGuard As Variant: Set scopeGuard = UsingTracer(MODULE_NAME, PROC_NAME)

    If (SomeFlag) Then Exit Sub ' ここで自動的に終了ログが記録される

End Sub ' ここで自動的に終了ログが記録される
```

:::message
**VBA 中級者・上級者に向けたコメント**
RAII（Resource Acquisition Is Initialization）の概念を参考にしています。
元々は C++ に由来し、最近だと Rust でも見かける設計思想です。

リソースの確保と解放をオブジェクトの生成と破棄に結びつける考え方です。

本ロガーでは、リソースの確保と解放の代わりに、スタックトレース情報の更新処理を担わせています。

VBA にはコンストラクタ（`Class_Initialize`）、および、デストラクタ（`Class_Terminate`）は存在します。  
しかし、引数を取るコンストラクタはサポートされていません。  
そのため、インスタンス生成と初期化を行うラッパー関数を用意して対応しています。
::: 

## シンプルな使用方法（コピペ操作がメイン）
具体例を挙げて解説します。

### スタックトレース用の記述
- step 1
モジュールの一番上に下記をコピペし、モジュール名をコピペする。

`Private Const MODULE_NAME As String = "MyModule"`

- step 2
関数の先頭行に下記をコピペし、`PROC_NAME As String = "SubProc"` の `"SubProc"` 部分を実際の関数名に置き換えます。

` Const PROC_NAME As String = "SubProc": Dim scopeGuard As Variant: Set scopeGuard = UsingTracer(MODULE_NAME, PROC_NAME)`

- 完了！

### 初期化/終了処理の記述
- step 1
前のセクション「このロガーで出来ること」で示した `Sub UseCase()` を関数ごとコピペします。
`Sub UseCase()` の様に、ロガーの初期化と終了処理を1つの関数にまとめ、メインの処理を初期化後に始める設計にすると、コピペでの流用がしやすいと思います。

- step 2
`Sub UseCase()` の前半に記述されている `With config` ~ `End With` の間にあるパラメータを編集する。
下記の様にロガーの挙動をカスタマイズ出来ます。

```vba
Dim config As LoggerConfigStruct
With config
    .IsIncludeCallerInfo = True ' スタックトレースの有無
    .IsWriteToImmediate = True ' イミディエイトウインドウに出力するか
    .IsWriteToExcelSheet = True ' エクセルシートに出力するか
    Set .OutputExcelSheet = ActiveSheet ' どのエクセルシートに書き込むか
End With
```

- 完了！

### ログ出力の記述
- step 1
`MyLogger.Log` をコピペして、以下のように記述します。

`MyLogger.Log "ログの内容"`

- 完了！

VBA 標準のログ機能 `Debug.Print "ログの内容"` と同じ使用感です。

ロガーの初期化処理さえ完了していれば、**プロジェクト内のどの標準モジュール、クラスモジュールでも `MyLogger.log` と書くだけ**です。

（アドイン参照と初期化により）`MyLogger` という特別なオブジェクトは**変数の宣言や代入をせずとも利用可能**になっています。

```vba
' これだけで良い
MyLogger.log "ログの内容"

' この様な操作は不要
Dim logger As VbaLogger
Set logger = New VbaLogger
logger "ログの内容"
```

:::message
**VBA 中級者・上級者に向けたコメント**
シングルトンパターンを採用しています。

ユーザーは Logger_SingletonManager のグローバル関数のみを経由してロガー（Logger_Facade クラス）にアクセス出来ます。

Logger_SingletonManager は、常に同一の Logger_Facade インスタンスが返されるよう設計されており、複数インスタンスの生成を防いでいます。
::: 

## ロガーの挙動をカスタマイズ（ログの出力先など）
ロガーの初期化にて、「ログ出力先」「スタックトレースの有無」を選択可能です。

:::message
**VBA 中級者・上級者に向けたコメント**
ログの出力はストラテジーパターンで実装しています。
Initialize()、WriteLog(logInfo)、Terminate() をインターフェースとして定義しています。
ユーザー定義型（Type）の変数 `logInfo` にはログを構成する情報（スタックトレース等）が収納されています。

ですので、txt ファイルへの出力機能の追加などを楽に出来る様に設計しています。
今後、別記事でコード設計を解説する予定なので、それと合わせれば各自で拡張する事も可能です。
:::

## アドインとして提供
このロガーは Excel のアドイン（.xlam）形式で配布しています。

**プロジェクトごとにコードをコピー＆ペーストする必要がありません。**

PC にアドインを保存し、VBE の参照設定で本アドインプロジェクトにチェックを入れることで使用可能になります。

アップデートもアドインを差し替えだけで反映できるため、複数プロジェクトで利用する場合も管理が簡単です。

内部的な実装には複数のクラスモジュールを使用し、オブジェクト指向プログラミング（object-oriented programming：OOP）で開発しています。

しかし、**ユーザーは難しい OOP の知識がなくても使える**よう工夫しています。

前述の通り、ユーザーは限られた関数のみを使用します。
**使用する必要が無いクラス・関数はアクセスそのものが不可能な様に設計しています。**

# 開発背景：なぜスタックトレース取得は困難だったのか
## VBA 固有の事情
前述の通り、**VBA では実行時にプログラムからコールスタック情報を取得する手段がありません**。

実行時エラーが発生しても、`Err.Number` や `Err.Description` でエラー番号・メッセージが分かるだけで、**どの関数を経由してここに来たか**は分からないです。

呼び出し履歴ウィンドウも**実行が止まっていなければ使えません**。

そのため、**エラー発生箇所そのもの**だけでなく、**そこに至るまでの具体的な呼び出し経路や、どのモジュールのどのプロシージャが関連しているのかを特定することが困難**です。


:::message
エラー発生時、VBE の GUI でエラーが発生した場所をハイライトする機能はあります。
しかし、この機能は GUI ベースであり、ログの様な記録は残りません。
GUI に依存したデバッグ環境で、文書記録が残せない点において、デバッグ能力不足を感じます。
:::

# 既存の解決策
VBAでスタックトレースに類する情報を得るための既存の手法やツールはいくつか存在しますが、**それぞれに特徴と課題があります**。

まずは、大まかな特徴を紹介します。

## 各方法の特徴
### 1 手作業によるログ埋め込み
`Debug.Print` を開発者がコードの任意の箇所に手動で挿入する最も原始的かつ直接的な方法。
特別な準備は不要ですが、手間は大きいです。

### 2 エラー伝搬による擬似スタックトレース
エラー発生時に、各プロシージャでエラーオブジェクトに自身の情報を付加して呼び出し元に再スロー。
最終的なエラーメッセージから呼び出し経路を限定的に推測します。
エラー発生時の情報収集に特化しています。

[stackoverflow - Is it possible to retrieve the call stack programmatically in VB6?](https://stackoverflow.com/questions/127645/is-it-possible-to-retrieve-the-call-stack-programmatically-in-vb6)

### 3 CallByNameラッパー
`CallByName` 関数を利用し、特定の関数呼び出し（主にオブジェクトのメソッドやプロパティアクセス）を、ログ出力機能を持つ共通のラッパー関数経由に書き換えます。

https://zenn.dev/sakanapan/articles/fc8406da72675c

### 4 VBE 操作によるコード自動挿入
VBE をプログラムで操作し、対象プロシージャの最初と最後にログ出力コードを自動的に挿入します。

http://ahox.jp/prog/2025-05_excel-2016-vba-qr-code-generate-and-logging-stack-trace.html


### 5 vbWatchdog
包括的なエラーハンドリングとデバッグ支援を提供する高機能な有償アドイン。
詳細なエラーレポート、リアルタイムの変数監視、保護されたコードでのエラーキャッチなど、高度な機能を備えています。
エラー発生時の情報収集に強みがあります。

## 各方法の比較
「技術的側面」と「コストやライセンスなどの非機能的側面」で比較してみました。

**どれも一長一短である**ことが分かるかと思います。

**企業における複数人チームでの大規模なプロジェクト**を想定した場合、以下の様な希望が出てくるかと思います。

- **階層情報を含む**スタックトレースを**コード中の任意の場所**で取得し、**挙動を詳細に把握したい**
- 導入方法や使用方法は**シンプル**であってほしい
    - **学習コストを抑える、コーディング規則として定めやすいため**
- **ライセンスが明記**され、**商用利用が許可**されていてほしい
- 要件に合わせて**機能を拡張したい**（ログの出力先、ログに含める要素など）
- ライブラリ自体の**保守が楽**であってほしい（**デバッグが簡単な設計**になっていてほしい）

しかし、**これら全てを満たす方法は無いと私は感じました**。

# 開発目的
前項の`既存の解決策`で述べた通り、私が理想とする要件を満たす手法・ライブラリは見つかりませんでした。

そのため、**それら課題を全て解決できるロガーが無いならば、自作しよう**と考えました。

定めた要件は以下の通りです。

- **スタックトレース（呼び出し階層）を取得**できる
- スタックトレースを含んだログを**コード中の任意の場所で出力**できる
- **導入が簡単**
- **学習コストを抑える**
    - 手作業が必要な工程は可能な限り**自動化**し、**ヒューマンエラーを防止**する
    - **シンプルなユーザーインターフェース**（使用方法）で実装する
        - VBA の標準ロガー `Debug.Print "ログの内容"` と**同じ呼び出し方**にする
- **アドイン本体のコードを変更せず**に、**ロガーの挙動をカスタマイズ**できる
    - **ログの出力先を柔軟に変更**できるようにする
- アドイン本体のコードを変更する際の負荷を減らす
    - **機能拡張、デバッグが簡単な設計**にする
        - **ログの出力先を他者が拡張**しやすくする
- **オープンソース（OSS） とする**

## ロガーの初期化
ロガーアドインの機能を利用するためには、まず初めにロガーを初期化する必要があります。

初期化処理では、**ログの出力先**や**スタックトレースの有無**といった、ロガーの動作をカスタマイズします。

### 使い方
まず、`LoggerConfigStruct` というユーザー定義型（複数の変数を 1 つにまとめたモノ）の変数を用意します。

次に、`LoggerConfigStruct` の中にある変数の値を設定していきます。

その後、`NewMyLogger.Initialize` 関数を呼び出すことで、設定が反映されたロガーが利用可能になります。

以下に、初期化処理の基本的な記述例を示します。

```vba
Sub UseCase()
    ' === ロガーの初期化 ===
    ' ロガーの動作を設定するための変数を作成する
    Dim config As LoggerConfigStruct

    ' config の中にある各変数に値を代入する
    With config
        ' スタックトレース情報を含むか（True/Flase）
        .IsIncludeCallerInfo = True
        ' イミディエイトウインドウへログを出力するか（True/Flase）
        .IsWriteToImmediate = True
        ' エクセルシートへログを出力するか（True/Flase）
        .IsWriteToExcelSheet = True
        ' ログを書き込みエクセルシートを指定（Worksheet 型）
        Set .OutputExcelSheet = ActiveSheet
    End With

    ' ロガーを初期化・設定を反映
    NewMyLogger.Initialize config
End Sub
```

```vba:エクセルシート指定の例
' VBA コードを実行した際に開いているエクセルシート
ActiveSheet

' VBA コードを実行した際に開いているエクセルファイル
' にて SheetName と名前が付けられたシート
' 一致するシートが存在しない場合はエラーになるので注意
ThisWorkbook.Sheets("SheetName")
```

:::message
**重要：初期化のタイミングと回数**
**ロガーの機能を利用する前に必ず初期化**をしてください。
（初期化をせずにログ出力やスタックトレース機能を使用した場合、エラーのポップアップが表示されます。）

メインとなる処理を始める前に、ロガーの初期化を実施することを推奨します。

また、VBA コード実行中にロガーの初期化を 2 回以上呼び出すことは非推奨です。
**基本的には、1 回のみ初期化を行ってください。**
:::


## スタックトレース機能
この機能を用いると、以下の2点が可能となります。

- プロシージャ（Sub、Function）の開始・終了のタイミングで自動的にログ出力

```
[2025-05-22 23:36:48.867][Trace] [MyModule.MainProc] >> Enter MainProc
[2025-05-22 23:36:48.898][Trace] [MyModule.MainProc] << Exit  MainProc
```

- 通常のログ出力に、スタックトレース情報を付与

```
[2025-05-22 23:36:48.875][INFO] [MyModule.SubProc < MyModule.MainProc] 処理2
```

スタックトレース機能を利用するには、監視したいプロシージャの先頭に特定の1行を追加するだけです。

### 使い方
プロシージャの冒頭に、以下の形式で 1 行記述します。

```vba
Private Const MODULE_NAME As String = "MyModule"

Sub MainProc()
    ' この1行を追加するだけで関数の開始・終了を記録できる
    Const PROC_NAME As String = "MainProc": Dim scopeGuard As Variant: Set scopeGuard = UsingTracer(MODULE_NAME, PROC_NAME)

    ' (プロシージャの処理)
End Sub
```

### 記述のポイント
#### 1 MODULE_NAME 定数の定義
各モジュール（標準モジュール、クラスモジュール）の冒頭で、そのモジュールの名前を文字列として定義します。

```vba
Private Const MODULE_NAME As String = "MyModule"
```

この `MODULE_NAME` は、そのモジュール内の複数のプロシージャで共通して使用します。

#### 2 PROC_NAME 定数の定義と UsingTracer の呼び出し
スタックトレースを記録したい各プロシージャの先頭行に下記を記述し、そのプロシージャ自身の名前を文字列として定義します。

```vba
Const PROC_NAME As String = "MainProc"
```

続けて、同じ行に下記を記述します。

```vba
: Dim scopeGuard As Variant: Set scopeGuard = UsingTracer(MODULE_NAME, PROC_NAME)
```

`scopeGuard` は Variant 型の変数として宣言します。
**この変数が、プロシージャの開始と終了の検知に重要な役割を果たします（詳細は後述）**。

`UsingTracer` 関数を呼び出し、引数として先ほど定義した `MODULE_NAME` と `PROC_NAME` を渡します。
**`UsingTracer` 関数により、開始ログが自動的に記録**されます。

:::message
**scopeGuard 変数の役割**
プロシージャが終了する際（`End Sub`、`Exit Sub`、`End Function`、`Exit Function`）、`scopeGuard` の機能により、そのプロシージャの終了ログが自動的に記録されます。

この仕組みにより、開発者は**スタックトレース情報が欲しいプロシージャに 1 行追加するだけ**で済みます。

**開始ログの指示を書けば必ず終了ログも自動的に行われる**ことが保証されるため、プロシージャの出口ごとに終了ログを記述する必要はありません。

**`scopeGuard` は手間を減らし、記述漏れによるスタックトレースの不整合を防ぐ、大切な変数です。**
:::


## ログの出力
プロシージャの実行中に任意の情報をログとして記録したい場合、`myLogger.Log` メソッドを使用します。

これは、VBA 標準の `Debug.Print` と同様のシンプルな使い方で、より高機能なログ出力を実現します。

### 使い方
ログを出力したい箇所で、以下のように記述します。

```vba
myLogger.Log "ここにログとして出力したいメッセージを記述します"
```

文字列だけでなく、変数やその組み合わせも出力可能です。

```vba
Dim count As Long
count = 10
myLogger.Log "現在のカウント: " & count & " です。"
```

タグを指定することも可能です（任意）。

```vba
myLogger.Log "Error タグ", LogTag_Error
myLogger.Log "Warning タグ", LogTag_Warning
myLogger.Log "タグ未指定"
```

```:ログの出力結果
[2025-05-23 00:40:55.381][ERROR] [MyModule.MainProc] Error タグ
[2025-05-23 00:40:55.384][WARNING] [MyModule.MainProc] Warning タグ
[2025-05-23 00:40:55.388][INFO] [MyModule.MainProc] タグ未指定
```

`myLogger.Log "タグを指定することも可能",` まで記入すると、タグの候補が表示されます。
（VBE のインテリセンス機能）

マウスクリック、または、矢印キーで選択してエンターでタグを選べます。

![ログ出力の命令時にオプションとしてタグを指定できる、タグの候補がインテリセンス機能で表示されている。](/images/5f91a2fd269ffd/5f91a2fd269ffd-2025-5-23.webp)

タグは以下の6種類から選択出来ます。

タグの指定が無い場合は `Info` が自動的に選択されます。

`Trace` はスタックトレース機能で開始・終了を自動でログ出力する際に使用されます。
通常のログ出力で指定することも可能ですが、非推奨です。

```vba
Public Enum LoggerLogTag
    LogTag_Debug = 0
    LogTag_Info = 1
    LogTag_Warning = 2
    LogTag_Error = 3
    LogTag_Critical = 4
    LogTag_Trace = 5
End Enum
```

## ロガーの終了処理
ロガーを使用した後は、適切な終了処理を行うことが重要です。

これにより、ロガーが使用していたリソース（例えば、Excelシートへの書き込み処理など）が正しく解放され、予期せぬ問題を防ぐことができます。

また、終了処理でエクセルの列幅の調整などの処理を行っています。

そのため、**全ての処理が終わった後、ロガーの終了処理も行ってください**。

### 使い方
`myLogger.Terminate` を呼び出します。
ロガーの終了処理が行われます。

```vba
myLogger.Terminate
```

# 活用例
## 再現困難な本番環境エラーの解析：ログベースのデバッグ
リリースした VBA マクロにて、**エンドユーザーの環境でのみエラーが発生**し、開発環境では**再現ができない**ケースは少なくありません。

本ロガーを活用することで、エラー発生時のスタックトレース（関数の呼び出し履歴）を含む詳細な実行ログを顧客環境から収集できます。

このログを解析することで、プログラムが実際にどのような順序で実行され、どの処理で問題が発生したのか、その**実行経路が可視化**されます。

例えば、「特定のデータパターン処理時にのみ発生するエラー」や「特定のユーザー操作時に発生するエラー」など、ログに残された処理の流れとデータ状態から、原因特定の手がかりを得やすくなります。

**これにより、運用保守の負担が減り、かつ、迅速な解決が期待できます。**


## チーム開発での情報共有と認識合わせ
チームで VBA 開発を進める際、**他のメンバーが作成したモジュールの挙動を正確に把握**したり、**発生した不具合の状況を的確に伝えたりする**ことは、時に困難を伴います。

特に、**複雑な処理が連携する箇所では、認識の齟齬が生じやすい**という課題があります。

本ロガーをチーム共通のデバッグ・情報共有ツールとして導入することで、プログラムの実行状況を客観的かつ詳細な記録として共有できます。

メンバーが遭遇した不具合を報告する際、スタックトレースを含むログを添付することで、**口頭説明の手間を無くし、断片的な情報共有による誤解を防げます**。

他にも、**コードレビューの効率向上**、ドキュメント（シーケンス図など）無しでも**担当外モジュールの挙動を理解しやすくなる**、といったメリットがあります。

- マクロでコードを直接的に自動編集されるのはリスクと感じた
    - 意図しないコード改変のリスク

        - 本ロガーの手間と大差が無くなってしまう

## 動作負荷・避けるべき使用方法
### スタックトレース機能
スタックトレース機能は関数ごとに 1 つ変数を生成します。

そのため、**For 文などで関数を数百回呼び出す、といった使い方をすると、予期せぬ負荷が生じる**可能性があります。

### ログ出力
ログの書き込み処理は、その都度実行されます。

高頻度でログ出力を行うと、その分、書き込み処理に時間を取られます。

そのため、**For 文でループ毎にログ出力するのではなく、10ループごとに1度、のように調整した方が良い**です。
