# はじめに：スタックトレースが欲しい理由と VBA の課題
プログラミングでデバッグを行う際、スタックトレース（関数の呼び出し履歴）は非常に重要な情報です。

エラーが起きたときに**どの関数からどの関数を経由してここに至ったのか**が分かれば、原因を突き止めやすくなります。

しかし、VBAでは**実行中にスタックトレースを取得する標準的な方法がありません**。
そのため、開発者は以下のような**手間のかかる方法**で対策する必要がありました。

- ブレークポイントを用意してコード実行を中断し、エディター（VBE）の GUI で確認する
- 自分で全関数にログ出力やエラー伝搬コードを仕込む
- 有償の商用アドイン `vbWatchdog` を導入する（$260、エラー時のみトレース可能）

**これは開発者にとって大きな負担であり、デバッグ作業の効率を下げる原因となります。**

本記事では、そうした VBA の課題を解決するために筆者が作成した VBA ロガーアドインを紹介します。
このアドインを使うと、**VBA でスタックトレースを含んだログ出力を簡単に行えます**。

:::message
**スタックトレース / コールスタック について**
厳密には、本ロガーはコード実行中のログ出力にコールスタック情報を含めることを可能とする、が正しいです。
本記事では、分かりやすさを優先し、一般的な用語であるスタックトレース呼びをしています。

- スタックトレース (Stack Trace)

エラー発生時の実行履歴を逆順に表示するもの。
- コールスタック (Call Stack)

関数の呼び出し情報を一時的に保管するメモリ領域。
:::

## VBA のコード
各クラス、関数の使い方は後ほど解説します。

```vba:標準モジュール
' === MyModule ===
Option Explicit
Private Const MODULE_NAME As String = "MyModule" ' モジュール名をここに書く

Sub UseCase()
    '=== ロガーの初期化 ===
    Dim config As LoggerConfigStruct ' ロガーの挙動を設定する
    With config
        .IsIncludeCallerInfo = True
        .IsWriteToImmediate = True
        .IsWriteToExcelSheet = True
        Set .OutputExcelSheet = ActiveSheet
    End With
    NewMyLogger.Initialize config  ' ロガーを初期化・設定を反映
    '=== 処理開始 ===
    MainProc
    '=== ロガーの終了処理 ===
    myLogger.Terminate ' ロガーの終了処理
    ReleaseMyLogger
End Sub

Sub MainProc()
    ' この1行を追加するだけで関数の開始・終了を記録できる
    Const PROC_NAME As String = "MainProc": Dim scopeGuard As Variant: Set scopeGuard = UsingTracer(MODULE_NAME, PROC_NAME)
    
    myLogger.Log "処理1"
    ' モジュール内の別関数を呼び出し
    SubProc
End Sub

Private Sub SubProc()
    Const PROC_NAME As String = "SubProc": Dim scopeGuard As Variant: Set scopeGuard = UsingTracer(MODULE_NAME, PROC_NAME)
    
    myLogger.Log "処理2"
    ' クラスモジュールの関数を呼び出し
    Dim hoge As MyClass
    Set hoge = New MyClass
    hoge.ClsMainProc
End Sub
```

```vba:クラスモジュール
' === MyClass ===
Option Explicit
Private Const MODULE_NAME As String = "MyClass"

Public Sub ClsMainProc()
    Const PROC_NAME As String = "ClsMainProc": Dim scopeGuard As Variant: Set scopeGuard = UsingTracer(MODULE_NAME, PROC_NAME)
    
    myLogger.Log "処理3"
End Sub
```


# ロガーの特徴と仕組み
このロガーアドインの主な特徴について説明します。

## 呼び出し階層（スタックトレース）の可視化
ログ出力時にどの関数がどれを呼び出したかが直感的に分かる形式で記録します。 

`MyModule.SubProc < MyModule.MainProc`
左が呼び出された関数（現在実行中の関数）、`<` の右側が呼び出し元の関数を示します。

**ネストした呼び出しも階層構造としてログから読み取れるため、複雑な処理の流れを追跡しやすくなります。**

## 関数の開始・終了を1行でトレース、終了は自動検出でミス防止
関数の冒頭に 1 行追加するだけで、スタックトレース情報が更新されます。

- 開始：追加した行の最後にある `UsingTracer` の呼び出しにより記録される
- 終了：現在の関数が終了（`End Sub`、`Exit Sub`）すると自動的に処理が走り記録される

**開始と終了が紐づいているので「開始は書いて終了を書き忘れ、階層がぐちゃぐちゃになる」といったヒューマンエラーを回避出来ます。**

```vba
Private Sub SubProc()
    ' この1行で SubProc の開始ログが記録される
    Const PROC_NAME As String = "SubProc": Dim scopeGuard As Variant: Set scopeGuard = UsingTracer(MODULE_NAME, PROC_NAME)

    If (SomeFlag) Then Exit Sub ' ここで自動的に終了ログが記録される

End Sub ' ここで自動的に終了ログが記録される
```

:::message
**VBA 中級者・上級者に向けたコメント**
RAII（Resource Acquisition Is Initialization）の概念を参考にしています。
元々は C++ に由来し、最近だと Rust でも見かける設計思想です。

コンストラクタで初期化処理（リソース確保）、デストラクタで終了処理（リソース解放）を行うことで、インスタンス生成時にリソースを確保し、インスタンスが破棄される（通常はスコープを抜けて参照がなくなる）タイミングで解放されることを保証する考え方です。

本ロガーでは、リソースの確保・解放の代わりに、スタックトレース情報の更新処理を担わせています。

VBA にはコンストラクタ（`Class_Initialize`）およびデストラクタ（`Class_Terminate`）は存在します。  
しかし、引数を取るコンストラクタはサポートされていません。  
そのため、インスタンス生成と初期化を行うラッパー関数を用意して対応しています。
::: 

## シンプルな使用方法（コピペ操作がメイン）
具体例を挙げて解説します。

### スタックトレース用の記述
- step 1
モジュールの一番上に下記をコピペし、モジュール名をコピペする。

`Private Const MODULE_NAME As String = "MyModule"`

- step 2
関数の先頭行に下記をコピペし、`PROC_NAME As String = "SubProc"` の `"SubProc"` 部分を実際の関数名に置き換えます。

` Const PROC_NAME As String = "SubProc": Dim scopeGuard As Variant: Set scopeGuard = UsingTracer(MODULE_NAME, PROC_NAME)`

- 完了！

### 初期化/終了処理の記述
- step 1
前のセクション「このロガーで出来ること」で示した `Sub UseCase()` を関数ごとコピペします。
`Sub UseCase()` の様に、ロガーの初期化と終了処理を1つの関数にまとめ、メインの処理を初期化後に始める設計にすると、コピペでの流用がしやすいと思います。

- step 2
`Sub UseCase()` の前半に記述されている `With config` ~ `End With` の間にあるパラメータを編集する。
下記の様にロガーの挙動をカスタマイズ出来ます。

```vba
Dim config As LoggerConfigStruct
With config
    .IsIncludeCallerInfo = True ' スタックトレースの有無
    .IsWriteToImmediate = True ' イミディエイトウインドウに出力するか
    .IsWriteToExcelSheet = True ' エクセルシートに出力するか
    Set .OutputExcelSheet = ActiveSheet ' どのエクセルシートに書き込むか
End With
```

- 完了！

### ログ出力の記述
- step 1
`MyLogger.Log` をコピペして、以下のように記述します。

`MyLogger.Log "ログの内容"`

- 完了！

VBA 標準のログ機能 `Debug.Print "ログの内容"` と同じ使用感です。

ロガーの初期化処理さえ完了していれば、**プロジェクト内のどの標準モジュール、クラスモジュールでも `MyLogger.log` と書くだけ**です。

```vba
' これだけで良い
MyLogger.log "ログの内容"

' この様な操作は不要
Dim logger As VbaLogger
Set logger = New VbaLogger
logger "ログの内容"
```

:::message
**VBA 中級者・上級者に向けたコメント**
シングルトンパターンを採用しています。

ユーザーは Logger_SingletonManager のグローバル関数のみを経由してロガー（Logger_Facade クラス）にアクセス出来ます。

Logger_SingletonManager は、常に同一の Logger_Facade インスタンスが返されるよう設計されており、複数インスタンスの生成を防いでいます。
::: 

## ロガーの挙動をカスタマイズ（ログの出力先など）
ロガーの初期化にて、「ログ出力先」「スタックトレースの有無」を選択可能です。

:::message
**VBA 中級者・上級者に向けたコメント**
ログの出力はストラテジーパターンで実装しています。
Initialize()、WriteLog(logInfo)、Terminate() をインターフェースとして定義しています。
ユーザー定義型（Type）の変数 `logInfo` にはログを構成する情報（スタックトレース等）が収納されています。

ですので、txt ファイルへの出力機能の追加などを楽に出来る様に設計しています。
今後、別記事でコード設計を解説する予定なので、それと合わせれば各自で拡張する事も可能です。
:::

## アドインとして提供
このロガーは Excel のアドイン（.xlam）形式で配布しています。

**プロジェクトごとにコードをコピー＆ペーストする必要がありません。**

PC にアドインを保存し、VBE の参照設定で本アドインプロジェクトにチェックを入れることで使用可能になります。

アップデートもアドインを差し替えだけで反映できるため、複数プロジェクトで利用する場合も管理が簡単です。

## 複雑なプログラムを隠蔽
内部的な実装には複数のクラスモジュールを使用し、オブジェクト指向プログラミング（object-oriented programming：OOP）で開発しています。

しかし、**ユーザーは難しい OOP の知識がなくても使える**よう工夫しています。

前述の通り、ユーザーは限られた関数のみを使用します。
**使用する必要が無いクラス・関数はアクセスそのものが不可能な様に設計しています。**


# 開発背景：なぜスタックトレース取得は困難だったのか
## VBA 固有の事情
前述の通り、**VBA では実行時にプログラムからコールスタック情報を取得する手段がありません**。

実行時エラーが発生しても、`Err.Number` や `Err.Description` でエラー番号・メッセージが分かるだけで、**どの関数を経由してここに来たか**は分からないです。

呼び出し履歴ウィンドウも**実行が止まっていなければ使えません**。

そのため、**エラー発生箇所そのもの**だけでなく、**そこに至るまでの具体的な呼び出し経路や、どのモジュールのどのプロシージャが関連しているのかを特定することが困難**です。


:::message
エラー発生時、VBE の GUI でエラーが発生した場所をハイライトする機能はあります。
しかし、この機能は GUI ベースであり、ログの様な記録は残りません。
GUI に依存したデバッグ環境で、文書記録が残せない点において、デバッグ能力不足を感じます。
:::

## 既存の解決策
### 1 手作業による場当たり的な対応とその限界
#### 1-1 ログの手動埋め込み
・手法
各関数の開始/終了時に `Debug.Print "関数名 開始"` のようなログを手動で記述する方法。

・課題
- 保守性の著しい低下
    - 関数名変更時のログ修正漏れ、コピペミスによる誤った関数名の記録
- 記述漏れのリスクと作業負荷
    - 全ての関数への徹底は非現実的で、抜け漏れが発生しやすい
    - 開始は記述したが、終了を記述し忘れるなど
- コードの肥大化と可読性悪化

・所感
- **手作業が多い**
- **記述ミスのリスクも高く、コード改変も難しくなる**
- **チーム開発の場合、ルールの作成・徹底が必要となる**

#### 1-2 エラー伝搬による擬似スタックトレース
・手法
各関数でエラーをキャッチし、呼び出し元に関数名を付加して再スローし、最終的なエラーメッセージに呼び出し履歴を含める方法。

・課題
- 全関数へのエラーハンドリング実装が必須となり、開発・保守コストが非常に高い
- 手間に対して得られる情報が限定的
- エラーが発生しないとスタックトレース情報が得られない

・所感
- **手作業が多い**
- **エラー発生個所は分かるが、エラー発生時にしか情報が得られない**

・参考資料
[stackoverflow - Is it possible to retrieve the call stack programmatically in VB6?](https://stackoverflow.com/questions/127645/is-it-possible-to-retrieve-the-call-stack-programmatically-in-vb6)

### 2 開発支援ツール・ライブラリの活用と残る課題
#### 2-1 MZ-Tools、Rubberduck などの汎用 VBA 開発支援ツール
・利点
エラー処理コードのテンプレート挿入などで一部手作業を軽減

・課題
- VBA の根本的な制約（実行時のスタック情報取得不可）を解決するものではない
- あくまで手作業の肩代わり

・所感
- **記述の複雑さ・冗長さに由来する、可読性・保守性の低下は解決しない**

・参考資料
https://www.mztools.com/

https://github.com/rubberduck-vba/Rubberduck

#### 2-2 その他のライブラリ
・現状
- 関数の呼び出し情報を出力するものは存在する

・課題
- 先述の手作業による擬似スタックトレースの実装補助に留まることが多い
- 関数呼び出しの**階層情報**を含むものは少ない
    - 関数の開始、終了をログ出力するのみであることが多い
        - **ログの前後を読んで呼び出し順を把握する手間がある**
- 配布方法がコードを直接公開しただけで、コードのコピー作業が発生することが多い
- **ライセンスが未表示**のライブラリも散見される
    - 業務にて導入するリスクとなり得る
- **機能拡張が難しい**設計（1つのモジュールのみで設計する等）であることが多い

### 3 高機能だが導入障壁のある専用ソリューション
#### 3-1 vbWatchdog
・利点
グローバルなエラーキャッチと詳細なスタックトレース取得が可能で、VBA コミュニティで高く評価されている

・課題
- 高価である（$260）
- エラー発生時のみトレース可能

・所感
- **有償であり、導入のハードルが高い**
- **エラー発生時にしかスタックトレース情報が得られない**

・参考資料
https://www.everythingaccess.com/vbwatchdog.asp

# 開発目的
前項の`開発の背景`で述べた通り、従来のVBA環境でスタックトレースに類する情報を得るためには、**多大な手作業と工夫を凝らす**か、または、**高価な専用ツールに頼る**しかありませんでした。

特に、エラー発生時だけでなく、**コード実行中の任意のタイミングで手軽に呼び出し階層を把握したい**というニーズに応えるものは限られていました。

そのため、**これらの課題を全て解決できるロガーが無いならば、自作しよう**と考えました。

また、せっかく作成するのですから、公開を念頭に置きました。
そこで、作って終わりではなく、**自分/他者が拡張/カスタマイズしやすい事**、**導入が簡単である事**も要件として定義しました。

更に、他者のライブラリやツールを使用する際の学習コストを下げる事も重要であると思ったので、**手作業が必要な工程は可能な限り自動化する事**、**シンプルなユーザーインターフェース（使用方法）にする事**も目的に含めました。
