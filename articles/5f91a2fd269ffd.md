# はじめに
プログラミングでデバッグを行う際、スタックトレース（関数の呼び出し履歴）は非常に重要な情報です。

エラーが起きたときに**どの関数からどの関数を経由してここに至ったのか**が分かれば、原因を突き止めやすくなります。

しかし、**VBAでは「コード実行中」にスタックトレースを取得する標準的な機能がありません**。

そこで、**VBA でスタックトレースを含むログ出力を簡単に行える**ロガーアドインを作成しました。

```:ロガーのデモ
[2025-05-22 20:24:35.891][Trace] [MyModule.MainProc] >> Enter MainProc
[2025-05-22 20:24:35.898][INFO] [MyModule.MainProc] 処理1
[2025-05-22 20:24:35.906][Trace] [MyModule.SubProc < MyModule.MainProc] >> Enter SubProc
[2025-05-22 20:24:35.922][INFO] [MyModule.SubProc < MyModule.MainProc] 処理2
[2025-05-22 20:24:35.938][Trace] [MyModule.SubProc < MyModule.MainProc] << Exit  SubProc
[2025-05-22 20:24:35.938][Trace] [MyModule.MainProc] << Exit  MainProc
```

# VBA におけるスタックトレースの課題
## スタックトレースとは
**スタックトレース**は、プログラムの**実行経路**（**どの関数からどの関数が呼ばれたか**）を示す情報です。

例えば、C# では `Environment.StackTrace` を用いると以下の様な表示がされます。

```:下側の関数は上側の関数から呼び出されたと読み解きます
at System.Environment.GetStackTrace(Exception e)
at System.Environment.GetStackTrace(Exception e)
at System.Environment.get_StackTrace()
at Sample.Main()
```

https://learn.microsoft.com/ja-jp/dotnet/api/system.environment.stacktrace?view=net-9.0



VBA ではこの機能が**存在しない**ため、**エラー原因の特定が困難**になります。

実際、複数の記事で **VBA の問題点としてスタックトレースが貧弱であり、エラー解決の手間が増大している点**が挙げられています。

https://speakerdeck.com/narazaka/vbanameteta

https://qiita.com/zakuroishikuro/items/81ddd7a33fb346aeb6a0

https://darumaexcel.uijin.com/aboutooa/vba_limit.html

## VBA 標準機能の3つの限界
### 限界1：実行中のコールスタック取得不可
他言語との比較。

- C#：`Environment.StackTrace`
- Python：`traceback.print_stack()`
- VBA：**機能なし**

https://learn.microsoft.com/ja-jp/dotnet/api/system.environment.stacktrace?view=net-9.0

https://qiita.com/podhmo/items/6657ae7b6ffbadc8f773


### 限界2：エラー発生時の情報不足
VBAで取得できる情報は主に以下の3つです。

- `Err.Number`：エラー番号
- `Err.Description`：エラーメッセージ
- エラー発生箇所のハイライト（GUI 上のみ）

**最も重要な「どの経路でここに来たか」が分かりません。**


### 限界3：デバッグ時の制約
VBA のエディター（VBE）には呼び出し履歴ウィンドウが存在しますが、以下のような**制約があります**。

- 実行停止時のみ利用可能
- ログとして記録不可
- 顧客環境では使用不可
- 事前のブレークポイント設定が必要

以下の記事では VBE でのスタックトレース情報の確認方法が紹介されています。

https://qiita.com/Kamo123/items/e4c39f25f44876cf9e2f


## 開発現場への具体的影響
### 不具合対応の長時間化
- エラー発生箇所の特定が困難
- どの処理経路で発生したか特定が困難

### チーム内情報共有の困難
- 不具合報告の曖昧さ
    - 口頭での説明に依存し、客観的な記録が残らない
    - 再現手順の説明が不正確になりがち

### 顧客環境での対応遅延
- VBE アクセス不可でデバッグが困難
- エラー発生時の詳細な実行状況を後から追跡できない
- 推測ベースでの修正によるリスク増大
- 顧客への説明・対応が長期化


# 既存の解決手法
VBA でスタックトレース相当の情報を得るため、これまで様々な手法が開発されてきました。

本項では、**5 つの手法の特徴**を紹介します。

## 1. 手作業ログ埋め込み
### 概要
各プロシージャに `Debug.Print` を開発者が手動で挿入する最も原始的かつ直接的な方法です。

```vba
Sub MainProc()
    Debug.Print "MainProc 開始"
    ' 処理内容
    Debug.Print "MainProc 終了"
End Sub
```

### 特徴
- ✅ 利点
    - VBA 標準機能のみで実現可能
    - 導入コスト・学習コストがゼロ
    - 出力内容を完全にコントロール可能
- ❌ 欠点
    - 関数追加時に手動でログコード追加が必要
    - ヒューマンエラー（書き忘れ・記述ミス）が頻発
    - 大規模プロジェクトでは膨大な手作業が必要

### 適用場面
- ✅ 小規模な一時的コード
- ❌ 大規模プロジェクト・チーム開発

## 2. エラー伝播による擬似スタックトレース
### 概要
エラー発生時に各プロシージャで自身の情報を追加して上位に伝播させる方法です。

https://stackoverflow.com/questions/127645/is-it-possible-to-retrieve-the-call-stack-programmatically-in-vb6

```vba
Sub MainProc()
    On Error GoTo ErrorHandler
    SubProc
    Exit Sub
ErrorHandler:
    Err.Description = "MainProc > " & Err.Description
    Debug.Print Err.Description
    Err.Raise Err.Number
End Sub

Private Sub SubProc()
    On Error GoTo ErrorHandler
    ' エラーが発生する処理
    Dim myValue As Integer
    myValue = 1 / 0
    Exit Sub
ErrorHandler:
    Err.Description = "SubProc > " & Err.Description
    Err.Raise Err.Number
End Sub
```

```:実行結果
MainProc > SubProc > 0 で除算しました。
```

:::details エラー伝播方法の技術的補足
`On Error GoTo ErrorHandler`
これにより、エラーが発生した際に `ErrorHandler` が実行されます。

`ErrorHandler:`
このコード以下に書かれた内容が、エラー時に実行されます。

`Err`
このオブジェクトにアクセスすることで、エラー発生時の情報を取得できます。

`ErrorHandler`、`Err` の詳細については下記公式リファレンスをご確認ください。

https://learn.microsoft.com/ja-jp/office/vba/language/reference/user-interface-help/err-object

https://learn.microsoft.com/ja-jp/office/vba/language/reference/user-interface-help/on-error-statement

:::

### 特徴
- ✅ 利点
    - VBA 標準機能のみで実現
    - エラー発生時に呼び出し経路を取得可能
    - 通常時のパフォーマンスへの影響が少ない
- ❌ 欠点
    - エラー時のみ有効（正常フローは追跡不可）
    - 全プロシージャにエラーハンドリングコードが必要
    - エラー処理コードの管理が煩雑

### 適用場面
- ✅ 小規模な一次的コード
- ❌ 正常フローのデバッグが必要な場面・大規模プロジェクト・チーム開発


## 3. CallByName ラッパー手法
### 概要
`CallByName` 関数を利用し、プロシージャ呼び出しをログ出力機能を持つラッパー関数経由で実行する方法です。

:::details CallByName とは？
文字列名を使用してプロパティやメソッドを呼び出す関数です。

```vba
CallByName オブジェクト, "メソッドの名前", 呼び出しの種類, [引数]
```

・公式リファレンス
https://learn.microsoft.com/ja-jp/office/vba/language/reference/user-interface-help/callbyname-function
:::

**サンプルコード、及び、実行結果は以下の記事より引用しました。**

https://zenn.dev/sakanapan/articles/fc8406da72675c

```vba
' Call CallSubByName(クラスのインスタンス, "プロシージャ名", "プロシージャ引数")
Call CallSubByName(vTest1, "Sub1Param", "p1")
```

```:実行結果
2023/09/18 15:37:09.572 START: TestTarget#Sub1Param(VbMethod)
パラメータ1つ p1
2023/09/18 15:37:09.576 FINISH: TestTarget#Sub1Param(VbMethod)
```

### 特徴
- ✅ 利点
    - 関数呼び出し時に自動でログ出力
    - 統一されたログ形式
    - 後付けでの既存コードへの導入が容易
- ❌ 欠点
    - CallByNameによるパフォーマンス劣化
    - 標準モジュールの関数では別途アダプタークラスが必要
    - 通常の関数呼び出しと記述方法が異なり可読性が低下
    - 階層構造のスタックトレース情報がない

### 適用場面
- ✅ クラスベースの設計が確立している場合
- ❌ 標準モジュール中心の既存コード
- ❌ パフォーマンスが重要な処理
- ❌ 階層構造のスタックトレース情報が必要な場合

## 4. VBE 操作による自動挿入
### 概要
VBA マクロを実行し、対象プロシージャの最初と最後にログ出力コードを自動的に挿入する方法です。

http://ahox.jp/prog/2025-05_excel-2016-vba-qr-code-generate-and-logging-stack-trace.html

https://github.com/ahox/StackTraceLib_VBA

### 特徴
- ✅ 利点
    - マクロ実行により一括でコード挿入が可能
    - 階層構造のスタックトレース情報を取得
    - 元のコード構造を大きく変更せずに導入
- ❌ 欠点
    - VBA プロジェクトへのプログラムによるアクセスが必要
        - 企業環境では制限される可能性がある
    - 意図しないコード変更のリスク
    - 新規関数作成の度にマクロ実行が必要

### 適用場面
- ✅ 個人開発環境での使用
- ✅ セキュリティ制約のない環境
- ❌ 企業環境での使用

## 5. vbWatchdog

包括的なエラーハンドリングとデバッグ支援を提供する高機能な有償アドインです。

https://www.everythingaccess.com/vbwatchdog.asp

- ✅ 利点
    - 詳細なエラーレポート、リアルタイム変数監視など高機能
    - 商用製品としての安定性と信頼性
    - インストール後すぐに利用可能
- ❌ 欠点
    - $260 の導入コスト
    - 豊富な機能を活用するための学習コスト
    - 主にエラー発生時の情報収集に特化
    - 商用利用時のライセンス管理が必要

### 適用場面
- ✅ 予算に余裕のあるプロジェクト
- ✅ 包括的なエラー管理が必要な場合
- ❌ 小規模開発・個人プロジェクト
- ❌ コード実行中にスタックトレース機能を得たい場合

# ロガーアドインの設計方針
## 企業チーム開発で求められる要素
企業での複数人チームによる大規模プロジェクトでは、以下が必要不可欠です。

- **階層情報を含む**スタックトレースを**コード中の任意の場所**で取得し、**挙動を詳細に把握**
- **シンプルな導入・使用方法**による学習コスト削減
- **明確なライセンス**
- **要件に応じた機能拡張**（出力先、ログ要素のカスタマイズ）
- **保守しやすい設計**（デバッグ容易性、拡張性）

しかし、これらすべてを理想的に満たす手法は存在しませんでした**。

**そこで「理想のロガーがないなら自作しよう」と考えました。**

:::message
本ロガーはあくまで私が使いたいロガーとして定義しています。
用途が限定的ですが、ご容赦ください。
:::

## 設計要件
### 要件1：スタックトレース機能
- プロシージャ開始・終了のトラッキング
- 呼び出し階層の動的管理
- 任意の場所でのスタックトレース付きログ出力

### 要件2：導入・学習コストの最小化
- アドイン形式による配布
- 標準ロガー `Debug.Print` と同じ感覚で使用可能
- 手作業工程の自動化（手間を最小化、ヒューマンエラーの防止）

### 要件3：オープンソース
- オープンソース（OSS）として公開
- MIT ライセンス採用

### 要件4：柔軟なカスタマイズ性
- 設定による出力先変更（ロガーのコードを改変せず）
- VBA での機能拡張を容易にする設計（ロガーのコードを改変）
- オブジェクト指向プログラミングによる拡張性向上

### 要件5：保守性の確保
- オブジェクト指向プログラミングによる責務分離
- 明確なインターフェース設計

:::message
**オブジェクト指向プログラミングとは**

VBA において、オブジェクト指向プログラミング（OOP）は馴染みのないワードかもしれません。
OOP はプログラムの可読性・保守性・拡張性を向上させるテクニックです。

難しい概念なので、ここでは理解出来なくても大丈夫です。

**具体的にどの様に活用するかは、本シリーズの 4 記事目（内部アーキテクチャ・設計思想）で解説します。**
:::
