# はじめに
プログラミングでデバッグを行う際、スタックトレース（関数の呼び出し履歴）は非常に重要な情報です。

エラーが起きたときに**どの関数からどの関数を経由してここに至ったのか**が分かれば、原因を突き止めやすくなります。

しかし、**VBAでは「コード実行中」にスタックトレースを取得する標準的な機能がありません**。

そこで、**VBA でスタックトレースを含むログ出力を簡単に行える**ロガーアドインを作成しました。

```:ロガーのデモ
[2025-05-22 20:24:35.891][Trace] [MyModule.MainProc] >> Enter MainProc
[2025-05-22 20:24:35.898][INFO] [MyModule.MainProc] 処理1
[2025-05-22 20:24:35.906][Trace] [MyModule.SubProc < MyModule.MainProc] >> Enter SubProc
[2025-05-22 20:24:35.922][INFO] [MyModule.SubProc < MyModule.MainProc] 処理2
[2025-05-22 20:24:35.938][Trace] [MyModule.SubProc < MyModule.MainProc] << Exit  SubProc
[2025-05-22 20:24:35.938][Trace] [MyModule.MainProc] << Exit  MainProc
```

# 活用例
## エラーハンドリング不在時のデバッグ支援
VBA 開発では、**エラーハンドリングが不十分な場合、エラー発生時の状況把握が困難**になるという大きな課題があります。

特に、VBA 標準機能では**エラー発生時のスタックトレース情報（どのプロシージャからどのプロシージャが呼ばれてエラーに至ったか）が提供されない**ため、原因究明に時間を要することが少なくありません。

通常では

- エラーハンドリングを適切に記述する
- `Debug.Print` を処理ごとに記述する

といった対策を行います。

しかし、これらは工数が多く、開発速度優先で後回し・一部のみ実装とされるケースもあります。
また、特に致命的なのは、**エラーが起こると想定していなかった場合、エラー発生時に得られる情報は限定的 ※ である点です**。

**本ロガーのスタックトレース機能を使用すれば、この VBA の弱点を手軽にカバーできます。**

:::details ※ 標準機能でエラー時に得られる情報の例
プロシージャ `ProcessStepC()` にて 0 除算を行ったとします。

コードを実行すると、以下の様にエラーを知らせるダイアログが表示されます。

デバッグボタンを押すと、エラーが発生した行が黄色にハイライトされます。

![エラー発生時のポップアップ](/images/bab9538507644c/bab9538507644c-2025-5-31.webp)

![デバッグボタンを押した後の画面](/images/bab9538507644c/bab9538507644c-2025-5-31_1.webp)

**---得られる情報は以上です。---**

致命的なのは、上記情報は全て エディター（VBE）の **GUI で表示されるだけ**であり、**テキストデータ（ログなど）は有りません**。

無論、**情報量も不足**しています。

- **どのタイミングでエラーが起きたのか？**
- **どの様な順序でプロシージが呼び出されていたのか？**

といった情報は分かりません。

**---他の言語と比べ、VBA では対策なしにエラーが発生した際の代償がとても大きいと言えます。---**

>他の言語の場合、エラー時にスタックトレース情報がコンソールにテキストとして出力され最低限の情報が担保されているパターンが多いです。
:::


### 活用イメージ
では、本ロガーを使用すると、この状況がどのように改善されるのか見ていきましょう。

- **シナリオ**
ループ処理の中で、アイテムの番号に応じて複数の異なる処理が呼び出されます。
**特定の処理 (`ProcessStepC`) では意図的にエラーを発生させます**。
このコード例では、イントロで示したような**エラーハンドリングなしの状態を再現しています**。

```vba
Option Explicit
Private Const MODULE_NAME As String = "MyModule"

Sub CheckLogger()
    ' === ロガーの初期化 ===
    Dim config As LoggerConfigStruct
    With config
        .IsIncludeCallerInfo = True
        .IsWriteToImmediate = True
        .IsWriteToExcelSheet = True
        Set .OutputExcelSheet = ActiveSheet
    End With
    NewMyLogger.Initialize config
    ' === ログの出力 ===
    ProcessLargeData
    ' === ロガーの終了処理 ===
    myLogger.Terminate
End Sub

Sub ProcessLargeData()
    Const PROC_NAME As String = "ProcessLargeData": Dim scopeGuard As Variant: Set scopeGuard = UsingTracer(MODULE_NAME, PROC_NAME)
    
    myLogger.Log "処理開始"
    Dim i As Long
    For i = 1 To 10
        ' 条件によって処理が分岐
        myLogger.Log "[処理開始] " & i & " 番目のデータ"
        ProcessStepA
        If i Mod 2 = 0 Then ProcessStepB
        If i Mod 3 = 0 Then ProcessStepC
        myLogger.Log "[処理終了] " & i & " 番目のデータ"
    Next i
    myLogger.Log "処理完了"
End Sub

Private Sub ProcessStepA()
    Const PROC_NAME As String = "ProcessStepA": Dim scopeGuard As Variant: Set scopeGuard = UsingTracer(MODULE_NAME, PROC_NAME)
    ' 処理
End Sub

Private Sub ProcessStepB()
    Const PROC_NAME As String = "ProcessStepB": Dim scopeGuard As Variant: Set scopeGuard = UsingTracer(MODULE_NAME, PROC_NAME)
    ' 処理
End Sub

Private Sub ProcessStepC()
    Const PROC_NAME As String = "ProcessStepC": Dim scopeGuard As Variant: Set scopeGuard = UsingTracer(MODULE_NAME, PROC_NAME)
    ' わざとエラーを発生させる
    Err.Raise 513
End Sub
```
#### 得られるログ

```:ログの内容
[2025-05-31 00:14:33.509][Trace] [MyModule.ProcessLargeData] >> Enter ProcessLargeData
[2025-05-31 00:14:33.568][INFO] [MyModule.ProcessLargeData] 処理開始
[2025-05-31 00:14:33.570][INFO] [MyModule.ProcessLargeData] [処理開始] 1 番目のデータ
[2025-05-31 00:14:33.572][Trace] [MyModule.ProcessStepA < MyModule.ProcessLargeData] >> Enter ProcessStepA
[2025-05-31 00:14:33.574][Trace] [MyModule.ProcessStepA < MyModule.ProcessLargeData] << Exit  ProcessStepA
[2025-05-31 00:14:33.577][INFO] [MyModule.ProcessLargeData] [処理終了] 1 番目のデータ
[2025-05-31 00:14:33.579][INFO] [MyModule.ProcessLargeData] [処理開始] 2 番目のデータ
[2025-05-31 00:14:33.582][Trace] [MyModule.ProcessStepA < MyModule.ProcessLargeData] >> Enter ProcessStepA
[2025-05-31 00:14:33.583][Trace] [MyModule.ProcessStepA < MyModule.ProcessLargeData] << Exit  ProcessStepA
[2025-05-31 00:14:33.585][Trace] [MyModule.ProcessStepB < MyModule.ProcessLargeData] >> Enter ProcessStepB
[2025-05-31 00:14:33.589][Trace] [MyModule.ProcessStepB < MyModule.ProcessLargeData] << Exit  ProcessStepB
[2025-05-31 00:14:33.591][INFO] [MyModule.ProcessLargeData] [処理終了] 2 番目のデータ
[2025-05-31 00:14:33.594][INFO] [MyModule.ProcessLargeData] [処理開始] 3 番目のデータ
[2025-05-31 00:14:33.597][Trace] [MyModule.ProcessStepA < MyModule.ProcessLargeData] >> Enter ProcessStepA
[2025-05-31 00:14:33.600][Trace] [MyModule.ProcessStepA < MyModule.ProcessLargeData] << Exit  ProcessStepA
[2025-05-31 00:14:33.602][Trace] [MyModule.ProcessStepC < MyModule.ProcessLargeData] >> Enter ProcessStepC
```

このテキストとして出力されたログから、VBA・VBE 標準機能では得られない様々な情報が分かります。

- **どのタイミングでエラーが起きたのか？**
    - 3 番目のデータの [処理終了] ログがないため、3 番目の処理でエラーが起こっていると推測できます。
    - `ProcessStepC` の開始ログ (`>> Enter ProcessStepC`) はありますが、終了ログ (`<< Exit ProcessStepC`) がないことから、`ProcessStepC` の処理途中でエラーが発生したと特定できます。

- **どの様な順序でプロシージが呼び出されていたのか？ (スタックトレース)**
    - エラー発生で処理が止まったログの最終行を確認すると、[MyModule.ProcessStepC < MyModule.ProcessLargeData < MyModule.CheckLogger] という呼び出し階層が記録されています。
    - `ProcessLargeData` → `ProcessStepC` という順序でプロシージャが呼び出された結果、エラーに至ったと追跡できます。

:::message
今回の例ではネストが浅いため恩恵が少なく見えますが、処理呼び出しのネストが深い複雑なプログラムでは、このスタックトレースがエラー原因特定のための重要な情報になるでしょう。
:::

- **直前までの処理状況**
    - 1 番目と 2 番目のデータの処理（[処理開始] から [処理終了] まで）はログ上でも正常に完了していることが確認できます。

    - 3 番目のデータについても、`ProcessStepA` は正常に実行されたことがログから分かります。

:::message
これにより、エラーが起きた `ProcessStepC` が、それ以前の処理（1、2 番目のデータ）でも呼び出されていたかどうか、といった有用な情報が得られます。

例えば、`ProcessStepC` が 3 番目のデータで初めて呼び出されたのであれば `ProcessStepC` 自体のロジックに問題がある可能性が高いと考えられます。

逆に、以前も呼び出されていて問題がなく、3 番目のデータでエラーになったのであれば、呼び出し時の状況（パブリック変数の変化、引数など）が原因である可能性を疑う、といった考察が可能になります。
:::

# 注意事項
## パフォーマンスに関する考慮事項
「ログ出力、特にスタックトレース情報の収集やExcelシートへの書き込みは、処理時間を要します。頻繁に呼び出されるループ内での myLogger.Log の多用は、全体のパフォーマンスに影響を与える可能性があります。」
「開発・デバッグ時には詳細なログ（例: LogTag_Debug）が役立ちますが、本番運用時にはログの量を絞る（例: LogTag_Info 以上のみ、あるいはエラー発生時のみに限定する）か、ロガーの設定で出力を制御する（例: IsWriteToImmediate = False, IsWriteToExcelSheet = False にする、またはログレベルによるフィルタリング機能が将来的に追加されればそれを利用する）ことを検討してください。」
開発環境と本番環境でのログレベル設定

## 初期化とTerminateの対応
Initialize/Terminateの未実行に注意
例外処理発生時でもTerminateが確実に実行されるような設計を推奨
