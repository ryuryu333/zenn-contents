# はじめに
**VBA といえば「古い」「手続き型」というイメージを持たれがちです。**

実際、OOP（オブジェクト指向プログラミング）的な設計や、デザインパターンを積極的に取り入れた VBA のコードに出会う機会は多くありません。

しかし、私は **VBA のように機能が限られた言語だからこそ、OOP のエッセンスを活用し、可読性と保守性に優れたモダンなコードを書くべき**だと考えています。

具体的な手法の一例として「**ビルダーパターン**」があります。

実際のコードで軽く比較してみましょう。

```vba:VBA
' 一般的な書き方
' -> 多くのプロパティがあると、設定漏れや順序の間違いが起きやすい
Dim customer As New Customer
customer.Name = "山田 太郎"
customer.Address = "東京都千代田区1-1-1"
customer.Email = "taro.yamada@example.com"
customer.Age = 45
customer.Rank = "Gold" ' ランクを設定
customer.IsActive = True ' アクティブ会員

' ビルダーパターンならこう書けます
Dim customer As Customer
Set customer = CustomerBuilder.Create("山田 太郎") _
    .WithAddress("東京都千代田区1-1-1") _
    .WithEmail("taro.yamada@example.com") _
    .WithAge(45) _
    .WithRank("Gold") _
    .IsActive() _
    .Build()
```

**どちらが「顧客オブジェクトを意図通りに組み立てている」と分かりやすいでしょうか？**

特に、上記のように**多くの設定項目を持つオブジェクトを生成する際**に、流れるような記述で、**安全かつ直感的にインスタンスを組み立てられる**のが「ビルダーパターン」の強みです。

また、保守性を高めるという観点でも有用な手法です（詳細は後述）。

---

本記事では、VBA 開発でビルダーパターンをどう活かすかを、**実際の設計・実装例を用いて紹介**します。
さらに、ビルダーパターンが最適解とは限らないことにも着目し、**類似手法や他言語との比較**も交えながら解説します。

**「こんな書き方があるんだ」という視点で読み、自身のプロジェクトに適した手法を見つけるきっかけになれば幸いです。**


# ビルダーパターンとは
ビルダーパターンは、**複雑なオブジェクトの生成処理を、分かりやすく・安全に行うための設計パターン**です。

私感ですが、以下の様なメリットがあります。

- 1 共通の構築手順で、異なる種類のオブジェクトを作成できる
- 2 構築手順を抽象化し、作成意図を明示できる
- 3 構築の流れと構築手段を分離し、責務が明確になる

```vba:具体例（VBA）
Set loggerA = builder.Create _
                     .EnableWriteToImmediate _
                     .Build

Set loggerB = builder.Create _
                     .EnableWriteToExcelSheet(ActiveSheet) _
                     .Build
```

# なぜ VBA でビルダーパターンが有用なのか
モダンな言語では「引数付きコンストラクタ」や「イミュータブルなオブジェクト」といった機能が備わっており、**ビルダーパターンに類似した機能を簡単に実装できます**。

そのため、ビルダーパターンは**以下の様な場合にのみ実装し、それ以外は言語機能で済ませる**といった対応をする場面が多いかと思います。

- 極端にコンストラクタの引数が多い
- 構築手順を柔軟に記載したい

しかし、VBA には**引数付きコンストラクタなどの仕組みが実装されていません**。
そのため、VBA ではオブジェクトの構築を**柔軟**、かつ、**安全**に行う手段が乏しいと言えます。

例えば、よくある設計として「構造体（Type）＋With 構文」や「プロパティ＋Setter」が使われます。
これらは、設定項目が増えるほど**可読性が下がり**、また、**設定漏れを「システム的に防ぐ」**ことは難しいです。

**この課題に対して、ビルダーパターンは明確な解決策になります**。

## クラス図・全体アーキテクチャ
この設計では、以下の様に各クラスで責務（行う事）が明確に区別されています。

| コンポーネント名      | 役割・説明                                                                 |
|-----------------------|------------------------------------------------------------------------------|
| User                  | ロガーを利用するエンドユーザー                                               |
| IConfigBuilder        | ロガーの設定を構築する方法を外部に提示する                                  |
| ConfigBuilder         | ユーザーの指示に従って設定を構築し、ConfigDTO に記録する                    |
| ConfigDTO             | ロガー設定のデータを保持する                                                |
| LoggerController      | ConfigDTO を読み込み、ロガーの動作を制御する                                |

>DTO（Data Transfer Object の略）、馴染みが無い方は構造体・Typeみたいなモノだと思えば大丈夫です（本記事の文脈においては）。

![クラス図](/images/6ae28bb043ffd9/6ae28bb043ffd9-2025-7-7.webp)

:::details UMLコード（PlantUML）
```

class User
```

## コードの構成
先ほどのクラス図の中で、**ビルダーパターン**の中核を担っているのが `IConfigBuilder` と `ConfigBuilder` です。

### IConfigBuilder
まず初めに、「インターフェース（`IConfigBuilder`）」という仕組みについて触れておきます。

:::details インターフェースとは
このトピックだけで1記事書けてしまうほど奥深いので、ここでは概要にとどめます。
**一部、簡略化のために厳密さを犠牲にしている点はご容赦ください**。

---

インターフェースとは、「**このクラスはどんな操作ができるか**」を定義するための機能です。

今回の設計では、`User` は `ConfigBuilder` に直接アクセスするのではなく、**`IConfigBuilder` を通して必要な操作のみにアクセスできる**仕組みになっています。

`IConfigBuilder` は「受付の窓口」のような役割だと言えます。

このようにすることで

- `User` が **知っておくべきこと**（設定方法）
- `User` が **知らなくてよいこと**（具体的な設定の構築方法や `ConfigBuilder` 独自の処理）

を分離できます。

将来、`ConfigBuilder` の中身を大きく変更しても、**`IConfigBuilder` で定義した操作方法が変わらなければ `User` 側のコードは一切修正不要**になります。

---

もしインターフェースがなかった場合、`User` は `ConfigBuilder` のすべての処理にアクセスできてしまい、**`ConfigBuilder` 内部構造の変更が `User` にまで影響してしまうリスクが発生します**。

`User` が `ConfigBuilder` の**どの処理を利用できるのか（利用させるのか）を制御する**、重要な役割がインターフェースにはあります。

>インターフェース（Interface）を表す接頭語として `I` がよく用いられます。本コードでは `ConfigBuilder` のインターフェースを `IConfigBuilder` と表現しています。
:::




## 具体的な使い方・サンプルコード
このロガーでは、次のようにビルダーを通じてインスタンスを組み立てることができます。

### ロガーの初期化と設定

```vba
Dim builder As LoggerConfigBuilder
Dim logger As Logger

Set builder = LoggerConfigBuilder.Create
builder.EnableWriteToImmediate ' イミディエイトウィンドウへの出力を有効化
builder.EnableTagFiltering _
    .Add(LogTag_Info) _
    .Add(LogTag_Debug) _
    .Apply
Set logger = builder.Build
```
