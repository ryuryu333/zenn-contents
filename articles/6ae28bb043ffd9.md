# はじめに
**VBA といえば「古い」「手続き型」というイメージを持たれがちです。**

実際、OOP（オブジェクト指向プログラミング）的な設計や、デザインパターンを積極的に取り入れた VBA のコードに出会う機会は多くありません。

しかし、私は **VBA のように機能が限られた言語だからこそ、OOP のエッセンスを活用し、可読性と保守性に優れたモダンなコードを書くべき**だと考えています。

具体的な手法の一例として「**ビルダーパターン**」があります。

実際のコードで軽く比較してみましょう。

```vba:VBA
' 一般的な書き方
' -> 多くのプロパティがあると、設定漏れや順序の間違いが起きやすい
Dim customer As New Customer
customer.Name = "山田 太郎"
customer.Address = "東京都千代田区1-1-1"
customer.Email = "taro.yamada@example.com"
customer.Age = 45
customer.Rank = "Gold" ' ランクを設定
customer.IsActive = True ' アクティブ会員

' ビルダーパターンならこう書けます
Dim customer As Customer
Set customer = CustomerBuilder.Create("山田 太郎") _
    .WithAddress("東京都千代田区1-1-1") _
    .WithEmail("taro.yamada@example.com") _
    .WithAge(45) _
    .WithRank("Gold") _
    .IsActive() _
    .Build()
```

**どちらが「顧客オブジェクトを意図通りに組み立てている」と分かりやすいでしょうか？**

特に、上記のように**多くの設定項目を持つオブジェクトを生成する際**に、流れるような記述で、**安全かつ直感的にインスタンスを組み立てられる**のが「ビルダーパターン」の強みです。

また、保守性を高めるという観点でも有用な手法です（詳細は後述）。

---

本記事では、VBA 開発でビルダーパターンをどう活かすかを、**実際の設計・実装例を用いて紹介**します。
さらに、ビルダーパターンが最適解とは限らないことにも着目し、**類似手法や他言語との比較**も交えながら解説します。

**「こんな書き方があるんだ」という視点で読み、自身のプロジェクトに適した手法を見つけるきっかけになれば幸いです。**


# ビルダーパターンとは
ビルダーパターンは、**複雑なオブジェクトの生成処理を、分かりやすく・安全に行うための設計パターン**です。

私感ですが、以下の様なメリットがあります。

- 1 共通の構築手順で、異なる種類のオブジェクトを作成できる
- 2 構築手順を抽象化し、作成意図を明示できる
- 3 構築の流れと構築手段を分離し、責務が明確になる

```vba:具体例（VBA）
Set loggerA = builder.Create _
                     .EnableWriteToImmediate _
                     .Build

Set loggerB = builder.Create _
                     .EnableWriteToExcelSheet(ActiveSheet) _
                     .Build
```

# なぜ VBA でビルダーパターンが有用なのか
モダンな言語では「引数付きコンストラクタ」や「イミュータブルなオブジェクト」といった機能が備わっており、**ビルダーパターンに類似した機能を簡単に実装できます**。

そのため、ビルダーパターンは**以下の様な場合にのみ実装し、それ以外は言語機能で済ませる**といった対応をする場面が多いかと思います。

- 極端にコンストラクタの引数が多い
- 構築手順を柔軟に記載したい

しかし、VBA には**引数付きコンストラクタなどの仕組みが実装されていません**。
そのため、VBA ではオブジェクトの構築を**柔軟**、かつ、**安全**に行う手段が乏しいと言えます。

例えば、よくある設計として「構造体（Type）＋With 構文」や「プロパティ＋Setter」が使われます。
これらは、設定項目が増えるほど**可読性が下がり**、また、**設定漏れを「システム的に防ぐ」**ことは難しいです。

**この課題に対して、ビルダーパターンは明確な解決策になります**。

## コードの構成
### ILoggerBuilder
#### インターフェースとは
`ILoggerBuilder` の先頭にある `I` ってなんだ？となっている方がいるかと思います。

これは「LoggerBuilder のインターフェース」という意味です（Interface の I）。

VBA において、インターフェースはマイナーだと思いますので、具体的なコード提示の前に概念を解説します。

**なお、このトピックだけで1記事書けてしまうほど奥深いので、ここでは概要にとどめます**。
**一部、簡略化のために厳密さを犠牲にしている点はご容赦ください**。

https://learn.microsoft.com/ja-jp/office/vba/language/reference/user-interface-help/implements-statement

---

インターフェースとは、「**このクラスはどんな操作ができるか**」を定義するための機能です。

今回の設計では、`User` は `LoggerBuilder` に直接アクセスするのではなく、**`ILoggerBuilder` を通して必要な操作のみにアクセスできる**仕組みにしています。

**言い換えると、`ILoggerBuilder` は「受付の窓口」のような役割です**。

このようにすることで

- `User` が **知っておくべきこと**（設定方法）
- `User` が **知らなくてよいこと**（具体的な設定の構築方法や `LoggerBuilder` 独自の処理）

を分離できます。

将来、`LoggerBuilder` の中身を大きく変更しても、**`ILoggerBuilder` で定義した操作方法が変わらなければ `User` 側のコードは一切修正不要**になります。

---

もしインターフェースがなかった場合、`User` は `LoggerBuilder` のすべての処理にアクセスできてしまい、**`LoggerBuilder` 内部構造の変更が `User` にまで影響してしまうリスクが発生します**。

`User` が `LoggerBuilder` の**どの処理を利用できるのか（利用させるのか）を制御する**、重要な役割がインターフェースにはあります。

>インターフェース（Interface）を表す接頭語として `I` がよく用いられます。本コードでは `LoggerBuilder` のインターフェースを `ILoggerBuilder` と表現しています。


#### コード解説
最初に `ILoggerBuilder` のコードを提示した後、文法の解説を行います。

```vba:VBA
' ILoggerBuilder.cls
Option Explicit

Public Function Create() As ILoggerBuilder
End Function

Public Function EnableWriteToImmediate() As ILoggerBuilder
End Function

Public Function EnableWriteToExcelSheet(ByVal targetSheet As Worksheet) As ILoggerBuilder
End Function

Public Function Build() As Logger
End Function
```

**インターフェースでは、どんな操作ができるか（どんなプロシージャを定義し、外部クラスへ公開するか）を記述します**。

**具体的な処理方法はここでは記述しません**。

:::message
`ILoggerBuilder` では「ログの出力先としてイミディエイトウインドウを有効化する」といったユーザーがしたい事のみを羅列していくイメージです。
どの様にして有効化した事を記録・管理・処理するかは `LoggerBuilder` で考えます。
:::

### LoggerBuilder
#### コード解説
`LoggerBuilder` のコード以下は通りです。

初見では複雑に見えかもしれませんので、**各要素を個別に解説していきます**。

```vba:VBA
' LoggerBuilder.cls
Option Explicit
Implements ILoggerBuilder

Private mLogger As Logger
Private mIsWriteToImmediate As Boolean
Private mIsWriteToExcelSheet As Boolean
Private mTargetSheet As Worksheet

Private Sub Class_Initialize()
    mIsWriteToImmediate = False
    mIsWriteToExcelSheet = False
    Set mTargetSheet = Notihing
End Sub

Private Function ILoggerBuilder_Create() As ILoggerBuilder
    Set mLogger = New Logger()
    Set ILoggerBuilder_Create = Me
End Function

Private Function ILoggerBuilder_EnableWriteToImmediate() As ILoggerBuilder
    mIsWriteToImmediate = True
    Set ILoggerBuilder_EnableWriteToImmediate = Me
End Function

Private Function ILoggerBuilder_EnableWriteToExcelSheet(ByVal targetSheet As Worksheet) As ILoggerBuilder
    mIsWriteToExcelSheet = True
    mTargetSheet = targetSheet
    Set ILoggerBuilder_EnableWriteToExcelSheet = Me
End Function

Private Function ILoggerBuilder_Build() As Logger
    If mLogger Is Nothing Then Err.Raise 513, "Create() が実行されていません。"
    mLogger.Initialize(mIsWriteToImmediate, mIsWriteToExcelSheet, mTargetSheet)
    Set ILoggerBuilder_Build = mLogger
End Function
```

##### Implements
まず、`Implements` キーワードを利用して、どのインターフェースを実装するかクラスであるかを宣言します。

```vba
Implements ILoggerBuilder
```

##### プロシージャの定義
次に、`Implements` で宣言したインターフェースに定義されているプロシージャを記述しています。

```vba
' （参考）ILoggerBuilder で定義した Build プロシージャ
Public Function Build() As Logger
End Function

' LoggerBuilder で Build プロシージャを定義
Private Function ILoggerBuilder_Build() As Logger
    ' 実際の Logger を作って返す処理
End Function
```

:::details 細かい文法を知りたい人用の解説
##### アクセス修飾子
インターフェース `ILoggerBuilder`は、外部クラスから利用される（参照される）ので Public にします。
`User` は `ILoggerBuilder.Build` という形式で利用します。

一方、インターフェースの実装クラス `LoggerBuilder` の `Build` は Private にします。
これにより、`User` が `LoggerBuilder.Build` と呼び出すことは不可能になります。
`User` が `Build` を使う際に `ILoggerBuilder` を利用することを強制する、とも言えるでしょう。

##### プロシージャ名
<インターフェース名>_<プロシージャ名> と記述するルールになっています。

https://learn.microsoft.com/ja-jp/office/vba/language/reference/user-interface-help/implements-statement

:::

##### メソッドチェーン
`ILoggerBuilder` で実装すべきと定義されたプロシージャを見ていきましょう。

`Create` などのプロシージャは返り値として `ILoggerBuilder` が指定されています。
実装時に、**自分自身のインスタンスを示す `Me` を結果として返すことで、`Builder.Create.EnableWriteToImmediate` のようなメソッドチェーンが記述可能**になります。

```vba
Private Function ILoggerBuilder_Create() As ILoggerBuilder
    Set mLogger = New Logger()
    Set ILoggerBuilder_Create = Me
End Function
```

##### ロガーの構築
次に、ロガーの構築に関する記述を行います。

ロガー本体と設定用の値を管理するためにメンバ変数を宣言します。

```vba
Private mLogger As Logger
Private mIsWriteToImmediate As Boolean
Private mIsWriteToExcelSheet As Boolean
Private mTargetSheet As Worksheet
```

コンストラクタ `Class_Initialize`（クラスをインスタンス化した際に実行される処理）にて、各変数の初期値を設定します。

```vba
Private Sub Class_Initialize()
    mIsWriteToImmediate = False
    mIsWriteToExcelSheet = False
    Set mTargetSheet = Notihing
End Sub
```

`Build` にてロガーのインスタンスを返します。
このステップでロガーの初期化処理を指示するなり、各パラメータの書き込みなどを行うとよいでしょう。

```vba
' 例1
' ロガーの初期化処理を呼び出す
' 引数として、ビルダーパタンでユーザーが指定した値を渡す
Private Function ILoggerBuilder_Build() As Logger
    mLogger.Initialize(mIsWriteToImmediate, mIsWriteToExcelSheet, mTargetSheet)
    Set ILoggerBuilder_Build = mLogger
End Function

' 例2
' ロガーインスタンスが持つプロパティに値を書き込む
Private Function ILoggerBuilder_Build() As Logger
    mLogger.IsWriteToImmediate = mIsWriteToImmediate
    mLogger.IsWriteToExcelSheet = mIsWriteToExcelSheet
    mLogger.TargetSheet = mTargetSheet
    Set ILoggerBuilder_Build = mLogger
End Function
```

なお、このままではユーザーが `Create` を呼ばずに直接 `Build` を呼び出すと `mLogger` が Nothing の可能性があります。
そのため、チェック機構を設けます。

```vba
Private Function ILoggerBuilder_Build() As Logger
    If mLogger Is Nothing Then Err.Raise 513, "Create() が実行されていません。"
    mLogger.Initialize(mIsWriteToImmediate, mIsWriteToExcelSheet, mTargetSheet)
    Set ILoggerBuilder_Build = mLogger
End Function
```

---

このようにして

`User`
-> `ILoggerBuilder`（ロガーの構築方法を定義、外部に提示）
-> `LoggerBuilder`（具体的な設定処理を実装、外部からは隠蔽）
-> `Logger`（実際に使うロガー本体）

という役割分担ができました。

# より堅牢な設計にするための工夫
実際に VBA でビルダーパターンを利用した際、工夫した点を紹介します。


## 呼び出し順序の制限（インターフェースの分割）
各プロシージャの返り値を使い分けることで、メソッドチェーンとして次に呼び出せるプロシージャを制限できます。

例えば

- ビルダーパターンの呼び出しを開始するプロシージャを `Start` のみに限定したい
- プロシージャ `SettingA` の後に呼べるプロシージャは `SettingB` のみに制限したい

といったケースを考えます。

必要な挙動を整理すると以下の様になります。

- `Builder.` とした際は `Start` のみが呼び出し可能
  - -> 専用のインターフェース `IBuilderStart` を用意する
- `SettingA.` とした際は `SettingB` のみが呼び出し可能
  - -> 専用のインターフェース `IBuilderAfterA` を用意する
- `Start.` や `SettingB` の後は `IBuilder` で定義されたプロシージャが呼び出し可能

コードに落とし込むと以下の様になります。

```vba
' インターフェス IBuilder
Public Function SettingA() As IBuilderAfterA
End Function

Public Function Build() As SomethingInstance
End Function


' インターフェース IBuilderStart
Public Function Create() As IBuilder
End Function


' インターフェース IBuilderAfterA
Public Function SettingB() As IBuilder
End Function


' 実装クラス
Implements IBuilder
Implements IBuilderStart
Implements IBuilderAfterA

Private Function IBuilderStart_Create() As IBuilder
    ' 処理省略
    Set IBuilderStart_Create = Me
End Function

Private Function IBuilder_SettingA() As IBuilderAfterA
    ' 処理省略
    Set IBuilder_SettingA = Me
End Function

Private Function IBuilderAfterA_SettingB() As IBuilder
    ' 処理省略
    Set IBuilderAfterA_SettingB = Me
End Function

Private Function IBuilder_Build() As SomethingInstance
    Dim result As SomethingInstance
    ' 処理省略
    Set IBuilder_Build = result
End Function
```

# 他手法（Setter/Type/With/配列渡し等）との比較
VBA で「複雑な設定が必要なオブジェクトを作りたい」場合、ビルダーパターン以外にも様々な方法があります。

ここでは、**主要な実装手法とビルダーパターンを比較**し、それぞれのメリット・デメリットを整理します。

## 1. Setter（プロパティ代入）

```vba
Dim logger As New Logger
logger.IsWriteToImmediate = True
logger.IsWriteToExcelSheet = False
logger.TargetSheet = Nothing
```

### デメリット
- 直接ロガーのプロパティを参照している
  - **ロガーを使う側**が**ロガー内部**の詳細（変数）を知る必要がある
  - ロガーの変数名を変えると**呼び出し側にも影響が及ぶ**
- 必須項目、任意項目などをシステム的に設定できない

設定項目が多いと可読性が下がる

必須項目の設定漏れを防げない

初期化順序の強制やバリデーションが難しい
