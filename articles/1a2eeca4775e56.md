# はじめに
GitHub Actions の CI/CD に Nix を利用する際、**バイナリキャッシュを活用することでビルド時間を短縮できます**。

キャッシュツールは複数存在しており、それぞれ特徴があります。

本記事では、3 種類の方法を「**実際のビルド時間**」で比較します。

- [cachix-action](https://github.com/cachix/cachix-action)
- [cache-nix-action](https://github.com/nix-community/cache-nix-action)
- [magic-nix-cache-action](https://github.com/DeterminateSystems/magic-nix-cache-action)

**どのツールが自身のプロジェクトに最適化判断する材料となれば幸いです**。

:::message alert
**検証条件によって適したツールは変わります**。
**あくまで私の検証ではこうだった、という参考程度にとどめてください**。
:::


# 対象読者

- GitHub Actions × Nix で CI を回しており、ビルド時間短縮を検討している方
- キャッシュを公開したくない／リポジトリ内完結で扱いたい事情がある方
- 反対に、複数ジョブ・複数リポジトリ・複数ランナー間でキャッシュを共有したい方
- 「どれを選べばよいか」の初期判断材料がほしい方

# 結論
ビルドを高速化したい、キャッシュを公開したくない場合は cache-nix-action。
キャッシュ量が多い、複数の環境を構築する場合、cachix-action の方が高速な可能性あります。

**基本的には cache-nix-action を利用し、必要に応じて cachix-action も比較検討する、といったスタイルが良いと思いました**。

# 1. 背景
## 1.1 Nix のビルドとキャッシュ
**Nix のビルドは、キャッシュにより最適化されています**。
背景知識として、簡略的にビルドの流れを紹介します。

`nix build` 等のコマンドを実行した際、`flake.nix` 等を元にし、ビルドレシピ（derivation）からビルド成果物（例：`treefmt`）がハッシュとして算出されます。

```nix:flake.nix
packages.default = buildEnv {
    name = "example";
    paths = [ pkgs.treefmt ];
};
```

<!-- cspell:disable -->
```bash
/nix/store/pnl4n8xandadzdp0jlnbkq4smkldprzs-treefmt-2.3.1
```
<!-- cspell:enable -->


**`treefmt` のバイナリを構築する前に、以下の順でビルド成果物が既にあるかを確認します**。

- ローカルの Nix ストアに同一パスがあるか
- なければ、設定されたバイナリキャッシュにあるか
- どちらにもなければ、一からビルド

**デフォルトのバイナリキャッシュは cache.nixos.org です**。
**nixpkgs の多くはここにキャッシュが存在するため、初回からビルドをスキップして高速になります**。

<!-- cspell:disable -->
```bash:ビルド時のログ例
these 464 paths will be fetched (265.13 MiB download, 1391.91 MiB unpacked):
  # ...
  /nix/store/pnl4n8xandadzdp0jlnbkq4smkldprzs-treefmt-2.3.1
  # ...
copying path '/nix/store/pnl4n8xandadzdp0jlnbkq4smkldprzs-treefmt-2.3.1' from 'https://cache.nixos.org'...
# ...
```
<!-- cspell:enable -->

一方で、**独自定義のパッケージは cache.nixos.org に存在しない**ため、初回はキャッシュを活用できません。

## 1.2 ローカル / GitHub Actions でのビルドの差異
「**ローカル環境**」では、独自定義のパッケージであっても、2 回目以降はローカルの Nix ストアでのキャッシュによりビルドが高速化されます。

しかし、「**GitHub Actions**」では、実行ごとにまっさらな環境が立ち上がるため、ローカルの Nix ストアによる再利用が効きません。

つまり、**独自定義のパッケージを含むプロジェクトでは、毎回ビルドが必要となり、CI/CD の実行時間が長くなる傾向にあります**。

## 1.3 バイナリキャッシュツール
Nix には cache.nixos.org 以外の手段でビルド成果物を保存・再利用する仕組みが用意されています。

**本記事では、GitHub Actions での Nix ビルドにフォーカスして、以下のキャッシュツールを比較します**。

- cachix-action
- cache-nix-action
- magic-nix-cache-action

:::message
**cachix、attic など、ローカルでのビルドにおいても利用できるキャッシュツールがあります**。
**本記事ではスコープ外としますので、詳細は参考資料をご覧ください**。
:::

## 1.4 最適なツールはどれか
[次項](#24-各ツールのスター数)でデータを提示しますが、スター数だけで判断するなら `magic-nix-cache-action` が良いとなるでしょう。

**しかし、本当にそうでしょうか**。

「**自分の環境にとってベストなツール（＝CI/CD が高速化できるツール）はどれか**」を知りたいと思った事が本調査のきっかけです。

**`nix build` と `ジョブ全体` の時間を評価軸とし、初回ビルドと 2 回目のビルドの時間を計測して、キャッシュツールを比較します**。

# 2. 各ツールの紹介
## 2.1 cachix-action

- 外部のバイナリキャッシュサービス（Cachix）を利用
- ローカルと GitHub Actions・複数リポジトリ・複数 Job でキャッシュを共有可能
- 無料枠あり
  - ただし、キャッシュは公開される
  - プライベートは有料
- cache.nixos.org に存在しないモノのみをキャッシュする

https://app.cachix.org/

https://github.com/cachix/cachix-action

https://zenn.dev/trifolium/articles/fc89526ce652fa

## 2.2 cache-nix-action

- GitHub Actions のキャッシュを利用
- デフォルト設定では `/nix`、`~/.cache/nix`、`~root/.cache/nix` がキャッシュ対象
- GitHub Actions Cache 上では大きな 1 ファイルとして保存される
  - e.g. cache name: `nix-${{ runner.os }}-${{ hashFiles('**/*.nix', '**/flake.lock') }}`
- Nix の環境全体をキャッシュ＆復元するイメージ（筆者の印象）

https://github.com/nix-community/cache-nix-action

## 2.3 magic-nix-cache-action

- 細かい設定無しで手軽に利用できる
  - 公式レポジトリより引用
  - `Use Magic Nix Cache, a totally free and zero-configuration binary cache for Nix on GitHub Actions.`
- キャッシュの挙動は cachix に似ているイメージ（筆者の印象）
  - cache.nixos.org に存在しないモノのみをキャッシュする
    - のはずだが...今回の検証では cache.nixos.org にあるファイルもキャッシュしていた
    - 詳細は考察に記載、解決策募集中

https://github.com/DeterminateSystems/magic-nix-cache-action

# 3. 検証方法
## 3.1 ビルドする環境
検証にあたり 2 種類の Nix 環境を用意しました。

- marp
  - `pkgs.marp` の用に pkgs 登録のパッケージをのみを利用する環境
- zenn
  - `importNpmLock` を利用して node_module を Nix で再現する環境
  - 公式バイナリキャッシュが活用されにくい環境

### 3.1.1 marp - pkgs.<name> のみを利用する環境
Marp CLI を利用して Markdown からスライドを静的作成するのに利用している環境です。

`pkgs.marp-cli` の様に nixpkgs のパッケージを利用します。
そのため、ビルドでは公式バイナリキャッシュ（cache.nixos.org）を活用できます。

```nix:flake.nix
{
  description = "Marp environment";

  inputs = {
    nixpkgs.url = "github:nixos/nixpkgs/nixpkgs-unstable";
    flake-utils.url = "github:numtide/flake-utils";
  };

  outputs =
    { nixpkgs, flake-utils, ... }:
    flake-utils.lib.eachDefaultSystem (
      system:
      let
        pkgs = nixpkgs.legacyPackages.${system};
      in
      rec {
        packages = {
          marp_build = pkgs.buildEnv {
            name = "marp_build_tools";
            paths = with pkgs; [
              marp-cli
              chromium
              noto-fonts-cjk-sans
            ];
          };
          default = packages.marp_build;
        };

        devShells = {
          default = pkgs.mkShell {
            nativeBuildInputs = with pkgs; [
              packages.marp_build
              gh
            ];
          };
        };

        apps = {
          marp = {
            type = "app";
            program = "${pkgs.marp-cli}/bin/marp";
          };
          default = apps.marp;
        };
      }
  );
}
```

### 3.1.2 zenn - node_module を nix で再現する環境
Zenn CLI を利用してローカル環境で記事を作成するのに利用している環境です。

Node.js のツールを利用するために、`importNpmLock` で node_module を Nix で構築します。

https://github.com/NixOS/nixpkgs/tree/master/pkgs/build-support/node/import-npm-lock

**`importNpmLock` によるビルドは公式バイナリキャッシュ（cache.nixos.org）が効きません**。
この点が、marp 環境との明確な違いです。

```nix:flake.nix
{
  description = "Zenn CLI environment";

  inputs = {
    nixpkgs.url = "github:nixos/nixpkgs/nixpkgs-unstable";
    flake-utils.url = "github:numtide/flake-utils";
  };

  outputs =
    { nixpkgs, flake-utils, ... }:
    flake-utils.lib.eachDefaultSystem (
      system:
      let
        pkgs = nixpkgs.legacyPackages.${system};
        npmRoot = ./node-pkgs;
        nodejs = pkgs.nodejs_24;
        inherit (pkgs) importNpmLock;
        npmDeps = importNpmLock.buildNodeModules {
          inherit nodejs npmRoot;
        };
      in
      rec {
        packages = {
          zenn_tools = pkgs.buildEnv {
            name = "zenn_tools";
            paths = with pkgs; [
              treefmt
              lychee
              npmDeps
            ];
          };
          default = packages.zenn_tools;
        };

        devShells = {
          zenn = pkgs.mkShell {
            nativeBuildInputs = [
              packages.zenn_tools
              pkgs.go-task
              importNpmLock.hooks.linkNodeModulesHook
            ];
            inherit npmDeps;
          };
          node = pkgs.mkShell {
            packages = [
              nodejs
              pkgs.npm-check-updates
            ];
          };
          default = devShells.zenn;
        };
      }
    );
}
```

## 3.2 ワークフロー
### 3.2.1 フローの内容について
#### ジョブ
マトリックス機能を利用して、ビルド環境 2 種 x（キャッシュツール 3 種 + ツール無し）＝ 8 パターンのジョブを定義しました。

#### トリガー
手動（`workflow_dispatch`）としました。

#### トリガー時に指定する変数
キャッシュ生成を含んだジョブの時間、キャッシュがある状態でのジョブ時間を比較したかったので、実行時  に `generate-cache` `use-cache` を選択し、目印としました。
処理に違いはなく、ジョブ名にのみ利用しています。

#### アクション
各ステップで利用しているアクションは各キャッシュツールの GitHub の README.md に記載されている利用例を模倣しました。

Nix のインストールがバラバラですが、統一はせずに公式の利用例に準拠した方法を優先しました。

#### コード

<!-- cspell:disable -->
```yml:build.yml
name: "build"

on:
  workflow_dispatch:
    inputs:
      phase:
        description: "Choose cache phase. Only used for naming the job."
        required: true
        type: choice
        options:
          - generate-cache
          - use-cache

jobs:
  build:
    strategy:
      matrix:
        cache_tool: [none, cachix-action, cache-nix-action, magic-nix-cache-action]
        target_nix_env: [marp, zenn]
    name: build_${{ matrix.target_nix_env }}_cachetool_${{ matrix.cache_tool }}_phase_${{ github.event.inputs.phase }}
    runs-on: ubuntu-latest
    concurrency:
      group: ${{ matrix.target_nix_env }}_${{ matrix.cache_tool }}_${{ github.event.inputs.phase }}_${{ github.ref }}
      cancel-in-progress: true
    defaults:
      run:
        working-directory: ${{ matrix.target_nix_env }}

    steps:
      - uses: actions/checkout@v5

      # === Install nix & setup cache tool ===
      # none
      - name: Install nix
        uses: cachix/install-nix-action@v31
        if: ${{ matrix.cache_tool}} == 'none'

      # cachix Action
      - name: Install nix
        uses: cachix/install-nix-action@v31
        if: ${{ matrix.cache_tool}} == 'cachix-action'

      - name: Setup cache tool
        uses: cachix/cachix-action@v16
        with:
          name: ryuryu333
          authToken: '${{ secrets.CACHIX_AUTH_TOKEN }}'
        if: ${{ matrix.cache_tool }} == 'cachix-action'

      # cache-nix-action
      - name: Install nix
        uses: nixbuild/nix-quick-install-action@v30
        with:
          nix_conf: |
            keep-env-derivations = true
            keep-outputs = true
        if: ${{ matrix.cache_tool}} == 'cache-nix-action'

      - name: Setup cache tool
        uses: nix-community/cache-nix-action@v6
        with:
          primary-key: nix-${{ matrix.target_nix_env }}-${{ runner.os }}-${{ hashFiles('**/*.nix', '**/flake.lock') }}
          restore-prefixes-first-match: nix-${{ matrix.target_nix_env }}-${{ runner.os }}-
          gc-max-store-size-linux: 1G
          purge: true
          purge-prefixes: nix-${{ runner.os }}-
          purge-created: 0
          purge-last-accessed: 0
          purge-primary-key: never
        if: ${{ matrix.cache_tool }} == 'cache-nix-action'

      # magic-nix-cache-action
      - name: Install nix
        uses: DeterminateSystems/nix-installer-action@v20
        if: ${{ matrix.cache_tool}} == 'magic-nix-cache-action'

      - name: Setup cache tool
        uses: DeterminateSystems/magic-nix-cache-action@v13
        if: ${{ matrix.cache_tool }} == 'magic-nix-cache-action'

      # === Build ===
      - run: nix build

      # === Run test command ===
      - run: nix develop -c zenn --version
        if: ${{ matrix.target_nix_env }} == 'zenn'

      - run: nix run .#marp -- --version
        if: ${{ matrix.target_nix_env }} == 'marp'
```

### 3.2.2 ワークフローの実行手順

- 手動で GitHub Actions の Cache を全て削除
  - `gh cache delete --all` を実行
- cachix の Cache を全て削除
  - Caches > ryuryu333 > Settings > Danger Zone > Clear をクリック
- 手動でワークフロー `build` を実行
  - `generate-cache` を選択
- 全てのジョブが完了するまで待機
- 手動でワークフロー `build` を実行
  - `use-cache` を選択
- 全てのジョブが完了するまで待機
- 最初に戻る

# 4. 検証結果
## 4.1 marp - pkgs.<name> のみを利用する環境
### 4.1.1 job 時間
キャッシュツールを利用しない条件（図左端）と比較して、job 全体の時間が減少したのは `cache-nix-action` だけでした。
ただし、短縮効果はごくわずかでした（36.5 sec -> 31.75 sec : 87%）。

`cache-nix-action` と `magic-nix-cache-action` はツール無しよりも時間が増加しました。

![marp job 時間一覧 use cache のみ](/images/1a2eeca4775e56/1a2eeca4775e56-2025-10-30_2.webp)

:::message
marp のビルドは殆どが公式バイナリキャッシュを利用するビルドであるため、キャッシュツールの恩恵が少ないのは予想通りと思われます。
キャッシュツールの導入によるオーバーヘッドで、job 時間が長くなったと思われます。

例外的に、`cache-nix-action` のみ job 時間が短縮されたのは何故でしょう？
job ではなく、粒度を細かくして build 時間を次は見てみます。
:::

### 4.1.1 build 時間
build 工程（RUN nix develop）を比較すると、`cache-nix-action` 以外は顕著な差がありませんでした。

![marp build 時間一覧](/images/1a2eeca4775e56/1a2eeca4775e56-2025-10-27_2.webp)

:::message
`cache-nix-action` のみ build 時間が短いのは、キャッシュ範囲の違いに起因すると思われます。

`cache-nix-action` は `nix/` 全体をキャッシュします。

>正確には、`nix/` 以外にもキャッシュ対象があり、また、`nix/` 以下であっても対象外のファイルがあります。詳細は[公式ドキュメントの Limitations](https://github.com/nix-community/cache-nix-action/tree/v6/?tab=readme-ov-file#limitations) を確認ください。

公式バイナリキャッシュにあるパッケージも含めて、ビルドによって生成された `nix/` 全体を GitHub Actions Cache に保存・復元していると言えます。

job のログを確認すると、キャッシュツールのアクションにて単一のファイル（645 MB）を GitHub Actions Cache からダウンロード・解凍しています。
そして、build では公式バイナリキャッシュから殆どダウンロードしていません。

# 5. 考察
まず cache-nix-action を第一候補に
理由: リポジトリ内で完結し、復元が速く、特に依存が大きいジョブで効果が大きい
小規模ジョブでも use-cache では安定して短縮（marp_build で約13%）
cachix-action は「共有幅」が広いと活きる
複数ジョブ/リポジトリ/ランナー/OS をまたいでキャッシュを共有したい場合に有効
今回の小規模ジョブではネットワーク往復などのオーバーヘッドが勝り、短縮が見えづらかった
大規模ジョブ（zenn_build）では有意な短縮（none 比で約1.6倍）
magic-nix-cache-action は今回は不利
本検証では generate/use いずれもオーバーヘッドが大きく、none と比べた優位性が見られなかった
導入容易さは魅力だが、現状の組み合わせ・条件では積極採用しづらいという結果
「build」と「job_total」を分けて見る意義
小規模ジョブではセットアップ/転送の固定コストが支配的になりやすい
大規模ジョブでは build が支配的になり、キャッシュが効いたときのリターンが大きい
運用上のヒント
初回（generate）は遅くて正常。定期的にキャッシュを温める運用（夜間やスケジュールトリガー）を併用すると安定する
cache-nix-action はキャッシュキー設計が重要（ロックファイルや nix flake の入力が変わるとミス）
cachix はシークレット管理、キャッシュの公開/私有方針とコストを事前に決める
まとめの指針
ビルド高速化かつキャッシュを公開したくない場合は「cache-nix-action」
キャッシュ量が多い・複数環境/リポジトリでの共有が重要なら「cachix-action」も検討
本検証条件では「magic-nix-cache-action」は不利だったが、環境次第で評価が変わる可能性はある
