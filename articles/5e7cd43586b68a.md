# はじめに
GitHub 連携によるバージョン管理や VSCode での記事作成に魅力を感じたので、セッティングしてみました。

普通に設定を進めると、Node.js や Zenn CLI をインストールしたり、VSCode（＋フォーマッター等の拡張機能）の設定を Zenn 仕様にしたり、と手間がかかります。
また、可能ならローカル環境はクリーンに保ちたいです。

そこで、Dev Containers を用いて Zenn 記事作成用環境を構築してみました。

主なメリットは以下の通りです。

- アプリの依存関係をコンテナ内に隔離し、ローカル環境をクリーンに保てる
- VSCode の設定や Markdown 関連の拡張機能を Zenn 執筆専用の環境として分離できる
- 環境構築の手順をコード (Dockerfile, devcontainer.json 等) で管理できる
- 異なるマシンでも同じ執筆環境を再現できる

## Dev Containers 設定ファイル作成
ローカルレポジトリ（筆者例:zenn_contents）直下に .devcontainer フォルダを作成します。
下記を参考に、.devcontainer フォルダ内に Dev Containers 用の設定ファイルを作成します。

```:フォルダ構成
zenn_contents/
├── .git/
└── .devcontainer/
    └── devcontainer.json
    ├── docker-compose.yml
    └── Dockerfile
```

### devcontainer.json
どの docker-compose.yml を使用するかを指定します。
VSCode で使用する拡張機能や設定を指定します。

```json:devcontainer.json
{
    "name": "zenn_contents",
    "dockerComposeFile": "docker-compose.yml",
    "service": "zenn_contents",
    //VScode接続時に開くフォルダを指定
    "workspaceFolder": "/work/zenn_contents",
    //VScodeの環境を指定
    "customizations": {
        "vscode": {
            //コンテナ作成時にインストールする拡張機能を記載
            "extensions": [
                "GitHub.copilot",
                "GitHub.copilot-chat",
                "negokaz.zenn-editor",
                "nodamushi.udon"
            ],
            //VScodeの設定を記載
            "settings": {
                //udon拡張の設定
                //ariclesディレクトリにある.mdファイルに画像を貼り付けた際
                //workspace/image/aritcle_name/に画像を保存
                //画像のリンクは相対リンクではなく、image/...で記載（zenn 仕様対応）
                "udon.baseDirectories": [
                    ["articles/*.md", "${workspaceFolder}/images/$fileBasenameNoExtension"]
                ],
                "udon.rule": [
                    ["articles/*.md", "![](/${relImage:${workspaceFolder}})"],
                    ["*.md", "![](${relImage})"],
                    ["*", "${relImage:${workspaceFolder}}"]
                ]
            }
        }
    }
}
```

### docker-compose.yml
ここと Dockerfile は普段の Docker と同じ書き方となります。
実は、フォルダのコンテナ:ローカル間共有は devcontainer.json でも設定できます。
個人的な好みで、Docker 側で設定できる箇所は Docker で行い、docker-compose.yml には VSCode 関連のみを記載する方針にしています。


```docker:docker-compose.yml
services:
  zenn_contents:
    build:
      dockerfile: Dockerfile
    tty: true
    # zenn用のフォルダ全てをマウント
    volumes:
      - ../:/work/zenn_contents/
```

### Dockerfile

```docker:Dockerfile
# 軽量なalpineイメージを選択
FROM node:22-alpine3.20
ENV NODE_PATH /opt/node_modules
# gitをコンテナ内で使用する際、ssh接続が必要なため、openssh-clientをインストール
RUN apk update && \
    apk add --no-cache git bash openssh-client
# zenn-cliのインストール
WORKDIR /work
RUN npm init --yes && \
    npm install -g zenn-cli@latest
# 実際に記事を管理するフォルダを作成
# コンテナとホスト間で共有するフォルダ
WORKDIR /work/zenn_contents
```

## SSH の設定
Zenn 記事の作成から公開まで一貫してコンテナ内で実施できた方が便利だと思います。
GitHub と SSH 接続にて通信している場合、コンテナ内で git push コマンドを使用するには、別途設定が必要となります。

https://code.visualstudio.com/remote/advancedcontainers/sharing-git-credentials#_using-ssh-keys

現状だと、コンテナ内で git push すると SSH 接続に用いる秘密鍵の情報がないとエラーになります。
Dev Containers ではローカルで実行中の ssh-agent が存在すれば、SSH 認証情報をコンテナ内で使用できる機能があり、こちらを利用します。

ssh-agent が自動起動する様に設定します。

```powershell:PowerShell
# SSHエージェントサービスを自動起動に設定
Set-Service ssh-agent -StartupType Automatic
# SSHエージェントサービスを開始
Start-Service ssh-agent
```

もしも、ssh-agent に GitHub 用の秘密鍵を登録していない場合、登録します。

```powershell:PowerShell
# .ssh直下の秘密鍵を登録、e.g. ~/.ssh/id_ed25519
ssh-add
# 特定の秘密鍵を登録
ssh-add ~/.ssh/GitHub/id_ed25519
```

# 使い方
## 初めて起動する際の操作
### コンテナの起動
VsCode で記事執筆用のディレクトリを開きます。
次に、コマンドパレットにて `Rebuild Container Without Cache and Reopen in Container` を選択します。

![](/images/5e7cd43586b68a/5e7cd43586b68a-2025-2-18_2.webp)

自動的に Docker コンテナのビルドが始まります。
右下に開発コンテナへ接続中という表記が消えるまで数秒待ちます。

### 拡張機能の再読み込み
初回のみ、拡張機能の読み込みが正常に行われないことがあります。
コンテナを出入りすると直ります。

コンテナから出るには、以下のいずれかを実施してください。
- ファイル＞最近使用した項目を開く＞記事執筆用のディレクトリを選択
- 画面左下の 青い><マーク をクリックし、コマンドパレットから `リモート接続を終了する` を選択

![](/images/5e7cd43586b68a/5e7cd43586b68a-2025-2-18.webp)

![](/images/5e7cd43586b68a/5e7cd43586b68a-2025-2-18_1.webp)

作成済みのコンテナに入る方法は以下の通りです。
- ファイル＞最近使用した項目を開く＞ディレクトリ名 [Dev Containeer] と書かれた方を選択
- コマンドパレットにて `Reopen in Container` を選択

### Zenn CLI の初期化
このままだと、ブラウザでのプレビュー機能が動かないので、コンテナ内で以下のコマンドを実行します。

```:sh
npx zenn init
```

https://zenn.dev/zenn/articles/install-zenn-cli

これにて執筆環境の構築は完了となります。

## 細かい操作方法について
### 画像の貼り付け
「Ctrl + Alt + V」で画像を貼り付けできます。

- 画像ファイルをコピペ
- スクリーンショットのペースト

どちらも対応しています。

md.ファイル内でのリンク形式は自動的に zenn 仕様`![](/images/article_name/image_name)`となります。

画像は下記フォルダに保存されます。

```:フォルダ構成
zenn_contents/
├── articles/
│   └── articles.md
└── images/
   └── article_name/
        └── image.webp
```

画像の貼り付けは udon 拡張機能によるものです。

## 環境構築で大変だった事
### 拡張機能が Active にならない！
`Rebuild Container Without Cache and Reopen in Container` でコンテナを再作成した際、zenn editor が Active になりませんでした。
`Reopen in Container` すると Active になるのですが、根本解決したわけではないのでしこりが残っています。

### 画像貼り付けと zenn 仕様のリンク
初期案では、画像の貼り付けを VSCode の標準機能で行うつもりで設定を書いていました。

https://code.visualstudio.com/docs/languages/markdown#_inserting-images-and-links-to-files

```json:devcontainer.json
{
    "customizations": {
        "vscode": {
            "settings": {
                // 旧設定
                "markdown.copyFiles.destination": {
                    "/articles/**/*": "images/articles/${documentBaseName}/"
                }
            }
        }
    }
}
```

この設定だと、.md に貼り付けした際にこの様になります。
`![](../images/~)`

これでは zenn 仕様に書き換える手間が発生してしまいます。
`![](/images/~)`

https://zenn.dev/zenn/articles/deploy-github-images

paste image 等の拡張機能もありますが、WSL 環境だと動作しません。
（＝ Windows ユーザーお断りになってしまい、よろしくないです）

https://zenn.dev/melon1891/articles/setting_for_zenn

なお、無理やり動かす方法もあったりします。
ただし、これだとセッティングが大変かもしれません。

https://another.maple4ever.net/archives/3457/
