# はじめに
**パッケージマネージャ Nix の利点の 1 つは対応しているパッケージの豊富さです**。

「最近流行っているツールをちょっと動かしてみたい」と思った時、大抵は Nix で使えます。
例えば、`OpenAI Codex CLI` を試したい時、以下のコマンドだけで実現できます。

```bash:Bash
nix-shell -p codex
```

利用可能なパッケージには Node.js に依存するモノ（e.g. `Zenn CLI`）も多く含まれます。
しかし、**マイナーなパッケージは pkgs 未登録な場合も多いです**。

----

**node2nix は Node.js パッケージを Nix で扱えるように変換するためのツールです**。

https://github.com/svanderburg/node2nix

node2nix には大きく分けて 2 つの使い方があり、それぞれ挙動が異なります。

- **package.json からプロジェクト全体を変換する方法**
- **パッケージリストからツールを個別に変換する方法**

本記事では、各方法の利用方法・使い分けを解説します。

  - pkgs に未登録のパッケージ含む

# 各方法の特徴
## A. プロジェクトベース（package.json + lockfile）
**`npm install` の挙動を Nix で再現する方法です**。

`package.json` と `package-lock.json` を元に、単一の node_modules を持つ derivation を生成します。

ESLint とそのプラグインのように、**パッケージ同士が互いを参照する必要がある場合に最適**です。

また、NPM の依存関係解決アルゴリズムで lockfile を作成し、**lockfile の依存関係を元に Nix ファイルへ変換するため、再現性が高い**です。

```bash:Bash
node2nix -l package-lock.json
```

```json:package.json
{
  "name": "node-eslint-env",
  "version": "1.0.0",
  "private": "true",
  "devDependencies": {
    "eslint": "9.34.0",
    "typescript-eslint": "8.41.0"
  }
}
```


## B. リストベース（個別パッケージ）
**NPM レジストリから、個々のパッケージの依存関係を取得し、Nix ファイルへ変換する方法です**。

指定した各パッケージが、それぞれ独立した derivation として生成されます。

`Zenn CLI` のような**単体で完結するツールを入れる場合に最適**です。

**設定がシンプルで、他のパッケージへの影響を気にせず追加・更新できます**。

```bash:Bash
node2nix -i node-packages.json
```

```json:node-packages.json
[
  { "eslint" : "9.34.0" },
  { "typescript-eslint" : "8.41.0" }
]
```

# 使い分け・ユースケース
## Case 1. 単体で完結するツールを入れたい
e.g. `Zenn CLI`
**-> B. リストベース（個別パッケージ）**

:::message
ツールごとに独立して管理できるため、設定がシンプルです。
devShell に追加するのも、pkgs からツールを追加するのと同じ感覚で扱えます。
:::

## Case 2. 互いに無関係な複数ツールを入れたい
e.g. `Zenn CLI`、`Codex CLI`
**-> B. リストベース（個別パッケージ）**

:::message
ツール間の依存関係がないため、個別に管理する事のメリットが大きいです。
片方のアップデートがもう片方に影響を与える心配がなく、トラブルの切り分けも容易になります。
:::

## Case 3. 本体とプラグインのように、相互参照が必要なツール群を入れたい
e.g. `ESLint`、`typescript-eslint`
**-> A. プロジェクトベース（package.json + lockfile）**

:::message
本体はプラグインを参照する必要があります。
`方法 A` ならば、`npm install` と同じ環境を再現してくれるため、依存関係の解決を node2nix に丸投げでき、確実かつ簡単です。

一方、`方法 B` でも相互の参照を結びつけることは可能ですが、`flake.nix` の記述が複雑になります。
:::

## Case 4. VSCode 拡張機能との連携が必要なツールを入れたい
e.g. `textlint`、`拡張 3w36zj6.textlint`
**-> A. プロジェクトベース（package.json + lockfile）**

:::message
VSCode `textlint 拡張` はプロジェクト直下の `node_modules` に `textlint` が存在することを期待します。

対策として `node_modules` のシンボリックリンクを作成する必要があります。

`方法 A` の方が容易にシンボリックリンクを設定できます。
:::


# 具体的な使い方：flake.nix への導入手順
ここからは、実際に node2nix を使って Node.js パッケージを flake.nix の devShell に導入するまでの具体的な手順を解説します。

どちらの方法も作業の流れは同じです。

- 1. node2nix で作成する Nix ファイルを保管する用のフォルダとして `node-pkgs` を作成
- 2. `node-pkgs` フォルダの中に json ファイルを作成
- 3. json ファイルに利用したい Node.js パッケージを記述
- 4. `node2nix` コマンドを実行
- 5. `default.nix`、`node-env.nix`、`node-packages.nix` が自動生成
- 6. `flake.nix` を記述
- 7. 完了

主な違いは、json の種類・書き方、`node2nix` のコマンド、`node-packages.nix` の中身、`flake.nix` の記述、です。

```フォルダ構成
your-repo
├─ flake.lock
├─ flake.nix
└─ node-pkgs
      ├─ default.nix
      ├─ node-env.nix
      ├─ node-packages.nix
      ├─ package.json        // 方法 A のみ
      ├─ package-lock.json   // 方法 A のみ
      └─ node-packages.json  // 方法 B のみ
```

### node2nix で Nix ファイルを作成

```bash:Bash
nix-shell -p nodejs_24 node2nix --run \

```


:::details コマンドの補足解説
上記コマンドは 2 つの処理を行っています。

- npm で `package.json` から `package-lock.json` を作成

```bash:Bash
nix-shell -p nodejs_24 --run 'npm install --package-lock-only --lockfile-version=2'
```

- node2nix で `package-lock.json` の依存関係を元に Nix ファイルを作成

```bash:Bash
nix-shell -p node2nix --run 'node2nix -d -l package-lock.json'
```

:::

### flake.nix を作成

```nix:flake.nix
{
  description = "Node pkgs environment";

  inputs = {
    nixpkgs.url = "github:nixos/nixpkgs/nixpkgs-unstable";
    flake-utils.url = "github:numtide/flake-utils";
  };

  outputs =
    { nixpkgs, flake-utils, ... }:
    flake-utils.lib.eachDefaultSystem (
      system:
      let
        pkgs = nixpkgs.legacyPackages.${system};
        my-node-pkgs = import ./node-pkgs/default.nix {
          inherit pkgs system;
          nodejs = pkgs.nodejs_24;
        };
        inherit (my-node-pkgs) nodeDependencies;
      in
      {
        devShells.default = pkgs.mkShell {
          packages = [
            nodeDependencies            
          ];
        };
      }
    );
}
```


:::details node_modules を必要とする場合
以下の場合、プロジェクトディレクトリ直下に `node_modules` が存在する必要があります。

- 一部のツール（e.g. `ESLint` + プラグイン）
- 一部の VSCode 拡張（e.g. `3w36zj6.textlint`）

次の様に `node_modules` のシンボリックリンクを作成することで対策できます。

```nix:flake.nix
in
{
  devShells.default = pkgs.mkShell {
    packages = [
      nodeDependencies            
    ];
    shellHook = ''
      ln -sfn ${nodeDependencies}/lib/node_modules ./node_modules
    '';
  };
}
```

仮に、シンボリックリンクを作っても正常に動作しない場合、以下の設定を試すと動くかもしれません。

```nix:flake.nix
in
{
  devShells.default = pkgs.mkShell {
    packages = [
      nodeDependencies            
    ];
    shellHook = ''
      ln -s ${nodeDependencies}/lib/node_modules ./node_modules
      export PATH="${nodeDependencies}/bin:$PATH"
      ln -s $NODE_PATH node_modules
    '';
  };
}
```

詳細は公式ドキュメントをご確認ください。

- `Creating a symlink to the node_modules folder in a shell session`

https://github.com/svanderburg/node2nix?tab=readme-ov-file#using-the-nodejs-environment-in-other-nix-derivations

# node2nix で Node.js ツールを Nix ファイルに変換する方法
## 主な方法、挙動の違い、使い分け

- A. node-packages.json を利用するパターン
  - 単体パッケージを Nix でビルド可能にしたい
  - 
  - 主眼: ツール箱として配る（例: textlint, eslint, prettier）
  - 特徴: lock 不要・差分が小さい・導入が軽い
  - 注意: プラグイン解決はできないため、textlint のプリセット参照は失敗（上記アラート参照）

- B. package.json と package-lock.json を利用するパターン: プロジェクト/プラグイン前提に最適
  - 主眼: npm の依存解決（lockfile）をそのまま再現し、プラグインや peerDeps を厳密固定
  - 特徴: Node 標準運用（npm scripts / audit 等）が可能
  - 注意: 一部ツールは物理的な `node_modules` を前提にするため、開発時は symlink を推奨（上記アラート参照）

クイック判断:
- 単体 CLI をサッと固定したい → A（node-packages.json）
- プラグインや peerDependencies 前提・npm 運用したい → B（package.json + lock）

### 補足: A でプラグインを使いたい場合のワークアラウンド
基本は B を推奨ですが、どうしても A で `textlint` のプリセットを使いたい場合は、`NODE_PATH` にプラグインの `lib/node_modules` を追加すると動く場合があります。

```nix:flake.nix
        devShells.default = pkgs.mkShell {
          packages = [
            my-node-pkgs.textlint
            my-node-pkgs."textlint-rule-preset-ja-technical-writing-12.0.2"
          ];
          shellHook = ''
            export NODE_PATH="${my-node-pkgs."textlint-rule-preset-ja-technical-writing-12.0.2"}/lib/node_modules:${NODE_PATH}"
          '';
        };
```

複数のプラグインを使う場合は、`:` 区切りで複数の `.../lib/node_modules` を連結してください。うまくいかない場合は、素直に B へ移行するのが確実です。
複数ツール＋プラグインを lockfile でカチッと再現したい: B（package-lock.json）
既存 Node プロジェクトをそのまま Nix 化したい: B

## A. node-packages.json を利用するパターン
単体のパッケージが使えればよい場合。
