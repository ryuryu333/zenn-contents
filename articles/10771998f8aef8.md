# はじめに
Power Automate Desktop では、すべての変数がグローバルに扱われるため、フローが大規模になるとどの変数がどの用途で使われているのか把握しにくくなることがあります。

ネット上で変数管理について語っている記事をあまり見かけないので、私が普段行っている方法を共有してみます。

:::message
これがベストプラクティスとは思っていないので「こんな管理方法もあるんだ～」程度で読んでいただければと思います。
:::

# 管理方法
## 案1 JSON形式で変数を論理グループ化する
変数をあらかじめ JSON 形式のオブジェクトとして定義し、性質や役割ごとにグループ化する方法です。

### ユースケースA ユーザー指定の定数を明示的にする
ユーザーが事前に手入力すべき定数的な変数、フロー中に動的に変わる変数、両者を区別するのに使えます。
例えば、他者にフローを共有する際に「変数 `UserInfo` の中身だけを書き換えて」と伝えるだけで済みます。

```json:UserInfo
{
  "User": {
    "Name": "山田", // 苗字のみを記入  この様にコメントも埋め込めます
    "ID": "12345" // 社員番号下5桁を記入
  },
  "PC": {
    "DataSaveDirectory": "C:/Users/username/Documents/projectX",
    "ID": "67890"
  }
}
```

### ユースケースB 関連性のある変数を一覧で確認しやくする
シェアポイントから特定のファイルを検索・ダウンロードする場合を考えてみます。

- FolderA の中身一覧を取得
- 対象のフォルダを検索（FolderB）
- FolderB の中身一覧を取得
- 対象のファイルを検索（FileX）
- FileX をダウンロード

といったフローになることが多いかと思います。

ファイルがダウンロード出来なかった場合、どこのアクションが想定外の挙動になっているでしょうか？

上記アクションの変数のみだったら、探すのは簡単です。
しかし実際のフローでは、別アクションの変数も一覧に表示されるので、骨が折れる作業になるでしょう。

![JSON にしていないとき](/images/10771998f8aef8/10771998f8aef8-2025-3-6_1.webp)

そこで、以下の様に JSON で各アクションの結果を集約します。

```json:SharePointDownload
{
  "Site": {
    "Path": "https://domain.sharepoint.com/sites/sitename",
  },
  "RootFolder": {
    "Path": "/Doclib1/projectX/data",
    "ID": "xxxxxxxxx",
    "Items": ""
  },
  "ChildFolder": {
    "Path": "",
    "ID": "",
    "Items": ""
  },
  "TargetFile": {
    "Path": "",
    "ID": ""
  }
}
```

フロー実行後に `SharePointDownload` をフロー変数一覧から開くと以下の様に表示されます。
`ChildFolder` の `Items` が空になっているので、この段階でデータが上手く取得できていないと簡単に判断できます。

### ユースケースC 機能拡張に伴い変数を追加する
事前に定数として設定する変数 `mail` を追加したい場合、通常の`変数の設定`アクションを用いる方法だと、アクション数とフロー変数が増加します。
JSON にしておけば 1 行追記するだけで済みます。

```json:UserInfo
{
  "User": {
    "Name": "山田",
    "ID": "12345",
    "mail": "address" //ここを追加
  },
  "PC": {
    "DataSaveDirectory": "C:/Users/username/Documents/projectX",
    "ID": "67890"
  }
}
```


## 案2 一時的な変数は「Tmp」命名を連続アクション限定で使用する

案1：
概要
一時的な値（例：あるアクションの出力結果を直後のアクションでのみ利用する場合）については、変数名を「Tmp」に統一するというルールを設定します。ただし、同じ変数を3回以上利用する場合は、用途に応じた意味のある名前（例：「Input_CustomerData」や「Calc_TotalAmount」など）に変更することで、変数の役割を明確にします。

メリット

シンプルさと即時性
直近の連続処理専用の変数であることがひと目で分かるため、フロー内で一時的な計算や中間結果を扱う際に、どれが一時変数なのかが明確になります。
読みやすさの向上
連続したアクション間での一時的なデータ転送と、複数回に渡って利用される主要な変数とで命名ルールを分けることで、フロー全体の可読性が向上します。
検討すべき点

連続性の定義
「連続したアクション」とは具体的にどの範囲か、分岐やループをまたぐ場合はどうするかなど、運用ルールを明確に文書化する必要があります。
運用ルールの統一
チーム全体でこのルールを徹底するために、命名規則やコードレビューのプロセスに組み込み、継続的なルールの見直しを行うことが求められます。

案2：JSON形式で変数を論理グループ化する
概要
変数をあらかじめJSON形式のオブジェクトとして定義し、性質や役割ごとにグループ化する方法です。たとえば、ユーザー情報を扱う場合、以下のような形式で変数をまとめます。

json
コピーする
{
  "User1": {
    "Name": "田中",
    "ID": "12345"
  },
  "User2": {
    "Name": "徳井",
    "ID": "67890"
  }
}
これにより、関連する変数が1つのオブジェクト内にまとまり、どの変数がどのグループに属しているかが明確になります。

メリット

論理的整理
複数の変数をグループ化することで、同じカテゴリに属するデータが一元管理され、全体の構造が理解しやすくなります。
拡張性
変数のグループに新たなキーや値を追加することで、将来的にデータ構造を柔軟に拡張できます。
検討すべき点

実装の複雑さ
Power Automate Desktop では、すべての変数がグローバル扱いであるため、JSON形式で管理するにはオブジェクトの作成、パース、アクセスのための追加処理が必要になります。これがフロー全体の複雑さを増す可能性があります。
参照記述の煩雑さ
JSON形式の場合、値にアクセスする際の記述が CustomObject['User1']['Name'] のように長くなるため、シンプルな変数参照と比べて扱いがやや煩雑になる可能性があります。
実際の運用における適用検討
適用可能なケース
案1
比較的小規模なフローや、一時的な中間値を扱うシンプルなシナリオでは、案1のルールは非常に有用です。直近のアクション間でのみ利用されるデータについて、すぐに使い捨てられるため、フロー内の変数の整理に役立ちます。

案2
複数の関連データを一元管理する必要がある大規模なフローや、チームでの開発の場合には、案2のJSON形式によるグループ化が役立ちます。これにより、論理的なまとまりができ、変数の意図や役割が明確になります。

注意点とハイブリッドアプローチ
フローの規模やチームの開発体制によって、どちらの方法が適しているかは異なります。場合によっては、案1のシンプルな命名規則と、案2の論理的グループ化を併用するハイブリッドなアプローチも検討できます。
どちらの場合も、運用ルールの明文化、レビュー体制、ドキュメントの整備が不可欠です。チーム内でルールを共有し、定期的に見直すことで、長期的な保守性を確保できます。
この記事では、Power Automate Desktop の変数管理において、連続処理専用の一時変数として「Tmp」を使う案と、JSON形式で論理的にグループ化する案の2つのアプローチを紹介しました。それぞれのメリット・デメリットを理解した上で、実際のシナリオに最適な方法を選択・組み合わせることで、効率的なフロー管理が実現できるでしょう。
