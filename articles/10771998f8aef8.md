# はじめに
Power Automate Desktop では、すべての変数がグローバルに扱われるため、フローが大規模になるとどの変数がどの用途で使われているのか把握しにくくなることがあります。

ネット上で変数管理について語っている記事をあまり見かけないので、私が普段行っている方法を共有してみます。

:::message
これがベストプラクティスとは思っていないので「こんな管理方法もあるんだ～」程度で読んでいただければと思います。
:::

# 管理方法
## 案1 JSON形式で変数を論理グループ化する
変数をあらかじめ JSON 形式のオブジェクトとして定義し、性質や役割ごとにグループ化する方法です。

### ユースケースA ユーザー指定の定数を明示的にする
ユーザーが事前に手入力すべき定数的な変数、フロー中に動的に変わる変数、両者を区別するのに使えます。
例えば、他者にフローを共有する際に「変数 `UserInfo` の中身だけを書き換えて」と伝えるだけで済みます。

```json:UserInfo
{
  "User": {
    "Name": "山田", // 苗字のみを記入  この様にコメントも埋め込めます
    "ID": "12345" // 社員番号下5桁を記入
  },
  "PC": {
    "DataSaveDirectory": "C:/Users/username/Documents/projectX",
    "ID": "67890"
  }
}
```

### ユースケースB 関連性のある変数を一覧で確認しやくする
シェアポイントから特定のファイルを検索・ダウンロードする場合を考えてみます。

- FolderA の中身一覧を取得
- 対象のフォルダを検索（FolderB）
- FolderB の中身一覧を取得
- 対象のファイルを検索（FileX）
- FileX をダウンロード

といったフローになることが多いかと思います。

ファイルがダウンロード出来なかった場合、どこのアクションが想定外の挙動になっているでしょうか？

上記アクションの変数のみだったら、探すのは簡単です。
しかし実際のフローでは、別アクションの変数も一覧に表示されるので、骨が折れる作業になるでしょう。

![JSON にしていないとき](/images/10771998f8aef8/10771998f8aef8-2025-3-6_1.webp)

そこで、以下の様に JSON で各アクションの結果を集約します。

```json:SharePointDownload
{
  "Site": {
    "Path": "https://domain.sharepoint.com/sites/sitename",
  },
  "RootFolder": {
    "Path": "/Doclib1/projectX/data",
    "ID": "xxxxxxxxx",
    "Items": ""
  },
  "ChildFolder": {
    "Path": "",
    "ID": "",
    "Items": ""
  },
  "TargetFile": {
    "Path": "",
    "ID": ""
  }
}
```

フロー実行後に `SharePointDownload` をフロー変数一覧から開くと以下の様に表示されます。
`ChildFolder` の `Items` が空になっているので、この段階でデータが上手く取得できていないと簡単に判断できます。

### ユースケースC 機能拡張に伴い変数を追加する
事前に定数として設定する変数 `mail` を追加したい場合、通常の`変数の設定`アクションを用いる方法だと、アクション数とフロー変数が増加します。
JSON にしておけば 1 行追記するだけで済みます。

```json:UserInfo
{
  "User": {
    "Name": "山田",
    "ID": "12345",
    "mail": "address" //ここを追加
  },
  "PC": {
    "DataSaveDirectory": "C:/Users/username/Documents/projectX",
    "ID": "67890"
  }
}
```


## 案2 一時的な変数は「Tmp」命名を連続アクション限定で使用する
一時的な値（例：あるアクションの出力結果を直後のアクションでのみ利用する場合）については、変数名を「Tmp」に統一します。

:::message alert
ただし、何でもかんでも Tmp とするとデバッグ時に処理をトレースできなくなります。
また、意図しない値の参照が発生するリスクもあります。
個人開発であっても、Tmp を使用するルールを厳格にし、使用は最低限にすべきです。
複数人での開発の場合、そもそも Tmp を使用しないという判断をすべきです。
~~Power Automate が public/private、スコープの概念、クラス・関数の概念、静的型付け、といった機能を実装してくれれば、楽なのに...~~
:::

### ユースケースA 一時的な計算や中間結果を扱う場合
ユーザーのメールアドレスから、`@` 手前の文字列を取り出し、`_` を削除する、というフローを考えてみます。 
通常ですと、以下の様に組むでしょう。

![Tmpを使わない場合](/images/10771998f8aef8/10771998f8aef8-2025-3-6_5.webp)

変数 `CroppedText` 直後のアクション`テキストの置換`でしか使用せず、この変数がフロー変数一覧に居座るのは邪魔なだけです。

そこで、本運用を適用します。

![Tmpを使う場合](/images/10771998f8aef8/10771998f8aef8-2025-3-6_6.webp)

このフローでは、`CroppedText` の代わりに `Tmp` を使用します。
また、`テキストの置換`の出力も `Tmp` に置き換え、直後に `UserInfo.UserName` に代入しています。

もしも、デバッグの都合で`テキストのトリミング`アクションの結果を確認したい場合は、以下の様にします。

![Tmpを使う場合 トレースを意識した場合](/images/10771998f8aef8/10771998f8aef8-2025-3-6_4.webp)
![詳細画面](/images/10771998f8aef8/10771998f8aef8-2025-3-6_7.webp)

:::message alert
この Tmp を用いた運用は、通常のプログラミングにおいては愚策中の愚策と言える方法です。

通常、変数には意味のある一意な名前を付けます。また、変数へ意味の異なる値の再代入は原則 NG です。
クラスや関数には必要な変数のみを定義し、クラスや関数外から見える（参照できる）変数は制限します。
しかし、Power Automate Desktop ではこの概念がありません。

全ての変数が表示されてしまうがゆえに、変数一覧画面を見やすくするには変数の数を削減せざるを得ないです。
苦肉の策といったところです。
:::


# 運用方法（アクションの設定方法）
## 案1 JSON形式で変数を論理グループ化する
JSON 形式の変数はそのままフローで使用できません。
そのため、`変数の設定`アクションで JSON を記述し、`JSON をカスタムオブジェクトに変換`アクションを実行します。

```json:UserInfo
{
  "User": {
    "Name": "山田",
    "ID": "12345",
    "mail": "address" //ここを追加
  },
  "PC": {
    "DataSaveDirectory": "C:/Users/username/Documents/projectX",
    "ID": "67890"
  }
}
```

![フロー全体](/images/10771998f8aef8/10771998f8aef8-2025-3-6_10.webp)
![JSONの記述](/images/10771998f8aef8/10771998f8aef8-2025-3-6_8.webp)
![カスタムオブジェクトへ変換](/images/10771998f8aef8/10771998f8aef8-2025-3-6_9.webp)

JAON の書き方はこちらの記事などを参照ください。
https://zenn.dev/hyung8/articles/4a50af486e5a06

変数へのアクセス方法は `%UserInfo.User.Name%` もしくは `%UserInfo['User']['Name']%` です。

値の検索・置換はフロー変数一覧から実施できません。
そのため、Power Automate Desktop 外で置換処理します。

- フロー全体を選択し、コピーする
- VSCosde や メモ帳 に貼り付ける
- 置換機能を用いて書き換え
- 文字列全体をコピーする
- Power Automate Desktop のフロー画面で貼り付けする

以下の画像では、12 行目の`変数の設定`アクションでの `%UserInfo.UserName%` を `%UserInfo.ID%` に置換処理しています。

## 案2 一時的な変数は「Tmp」命名を連続アクション限定で使用する
管理方法のユースケースで述べた通りです。
アクションの出力変数の名前を Tmp に変更するだけです。

なお、入力と出力どちらも Tmp とすることも可能です。
（Tmp の値を使用して何らかの処理をした後、結果を Tmp に代入されます）

Tmp を使用するのは
**あるアクションの出力結果を直後のアクションでのみ利用する場合**
に限定するのをお勧めします。

# 考案に至った経緯
約 400 アクションとなるフローを作成する過程で
・デバッグ時に変数の中身のモニタリング
・機能拡張で変数を追加
・リファクタリング
といった際に変数の数が多いと、何がどの変数であるのか分からなくなりました。

通常のプログラミングだと、関連性の高い変数は構造体や辞書型として定義できます。
```cs:C#
// 構造体
public struct User
{
    public string Name { get; set; }
    public int ID { get; set; }

    /// <param name="name">苗字のみを入力</param>
    /// <param name="id">社員番号下6桁を入力</param>
    public User(string name, int id)
    {
        Name = name;
        ID = id;
    }
}

// 辞書型
var user = new Dictionary<string, int>
{
    {"Name", "山田"},
    {"ID", 123456}
};
```

しかし、Power Automate Desktop にはこれら機能は存在しません。

代替案としては
- DataTable
- JSON（カスタムオブジェクト）
が使えそうでした。

拡張性や編集操作性が良好と感じた JSON を採用しました。


次に、1度しか使っていない変数に目を向けました。

通常のプログラミングでは、一時的な変数の存在はあまり問題になりません。
VSCode などの高性能エディターでは、デバッグ時に変数の数が多くても、注目したい変数を簡単に確認できます。
（コード中の変数をマウスオーバーすると値が表示されるなど）

また、ローカル変数はスコープ（例えば関数の実行）が終了すれば、自動的にデバッグ画面の変数ビューから表示されなくなります。

下記コードの場合、関数 `main()` の途中でで関数 `TrimmingNameFromMail()` が実行されます。
実行中は関数 `TrimmingNameFromMail()` 内に定義された変数はデバッグ画面に表示されます。
関数の処理が終了すると表示されなくなります。

```cs:C#
public class Hoge
{
    private string name { get; }
    private readonly string mail = "yamada_tarou@example.com";

    public void main(){
      // 何かしらの処理
      string result = TrimmingNameFromMail(mail);
      name = result;
      // 何かしらの処理
    }

    private string TrimmingNameFromMail(string mail){
      // 何かしらの処理
    }
}
```

また、似たような仕組みで、`if` 文の中で定義した変数は、`if` 文の途中はデバッグ画面の変数ビューに表示され、`if` 文が完了すると消えます。

しかし、Power Automate Desktop にはこれら機能は存在しません。
全ての変数がフロー変数に表示されます。

デバッグ機能の拡張は難易度が高いため、変数そのものを減らす方針で考え、Tmp の運用を思いつきました。

# おわりに
この記事では、Power Automate Desktop の変数管理において、JSON形式で論理的にグループ化する案と、連続処理専用の一時変数として「Tmp」を使う案、の2つのアプローチを紹介しました。

Tmp は功罪併せ持つ（なんならリスクの方が大きい）ので、採用は慎重になるべきです。
一方、JSON はリスクと比べて利点が大きいと思いますので、私は大半のフローで活用しています。

それぞれのメリット・デメリットを理解した上で、実際のシナリオに最適な方法を選択・組み合わせることで、効率的なフロー管理が実現できるでしょう。
